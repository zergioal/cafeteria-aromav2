1. ¿Qué es una base de datos?

Analogía: La cafetería sin sistema
Imagina que llevas la cafetería Aroma con cuadernos. Un cuaderno para los productos, otro para los clientes, otro para los pedidos. Cada vez que alguien pregunta "¿cuánto vendimos hoy?" tienes que revisar decenas de páginas manualmente. Cuando un precio cambia, debes borrarlo y reescribirlo en cada lugar donde aparece. Se pierden páginas, hay errores, es lento.

Una base de datos es como tener ese sistema de cuadernos convertido en un archivero digital perfectamente organizado: puedes buscar cualquier dato en segundos, actualizar en un solo lugar y obtener reportes automáticamente.

En términos técnicos, una base de datos es un sistema organizado para guardar, buscar, actualizar y eliminar información de manera eficiente y confiable. En una cafetería real necesitamos registrar:

  • Productos: nombre, precio, descripción, categoría, si está disponible o no.
  • Clientes: nombre, teléfono, correo electrónico, dirección de entrega.
  • Pedidos: qué productos pidieron, cuándo, cuánto pagaron, estado del pedido.
  • Usuarios del sistema: admins, cajeros, y sus permisos.

Tipos de bases de datos

Tipo Nombre técnico Cómo organiza los datos Ejemplos
Relacionales SQL Tablas con filas y columnas, relacionadas entre sí MySQL, PostgreSQL, SQLite, SQL Server
No relacionales NoSQL Documentos, colecciones, grafos, clave-valor MongoDB, Firebase, Redis, Cassandra

En el proyecto Aroma

Usamos PostgreSQL (relacional) a través de Supabase. Elegimos una base de datos relacional porque los datos de una cafetería tienen relaciones claras: un pedido pertenece a un cliente, un producto pertenece a una categoría. Las bases relacionales están diseñadas exactamente para eso.

2. Tablas, Columnas y Filas

Analogía: Como una hoja de Excel
Una tabla de base de datos es exactamente como una hoja de cálculo en Excel. Cada columna define un tipo de dato (nombre, precio, fecha). Cada fila es un registro específico (un producto concreto). La diferencia es que la base de datos es mucho más poderosa, veloz y segura.

Así se ve la tabla products de Aroma con datos reales:

TABLA: products
┌────────────────────┬─────────────────────┬─────────┬──────────────┬────────┐
│ id (UUID)          │ nombre              │ precio  │ categoria_id │ activo │
├────────────────────┼─────────────────────┼─────────┼──────────────┼────────┤
│ a1b2-c3d4-e5f6...  │ Café Americano      │ 12.00   │ cat-001      │ true   │
│ b2c3-d4e5-f6a7...  │ Café Latte          │ 18.50   │ cat-001      │ true   │
│ c3d4-e5f6-a7b8...  │ Jugo de Naranja     │ 15.00   │ cat-002      │ true   │
│ d4e5-f6a7-b8c9...  │ Pastel de Chocolate │ 25.00   │ cat-003      │ false  │
└────────────────────┴─────────────────────┴─────────┴──────────────┴────────┘

Cada elemento tiene su rol:

  • id: identifica de forma única a cada producto (ninguno puede tener el mismo id).
  • nombre: el texto con el nombre del producto.
  • precio: número decimal con exactamente 2 decimales.
  • categoria_id: referencia a qué categoría pertenece (Cafés, Jugos, etc.).
  • activo: si el producto está disponible en el menú o no.
Tip: ¿Por qué "activo" en vez de eliminar?
En sistemas reales casi nunca eliminamos registros definitivamente. Si "eliminamos" un producto que aparece en pedidos antiguos, esos pedidos quedarían rotos. En cambio, marcamos el producto como activo = false y simplemente no lo mostramos en el menú. Los pedidos históricos siguen intactos.

3. Tipos de datos en PostgreSQL

Cada columna de una tabla tiene un tipo de dato que define qué clase de información puede guardar. Elegir el tipo correcto es importante para la precisión y el rendimiento.

Tipo Para qué sirve Ejemplo en Aroma
UUID IDs únicos globales de 128 bits id = 'a1b2-c3d4-e5f6...'
TEXT Texto de longitud variable, sin límite nombre = 'Café Latte'
VARCHAR(n) Texto con máximo n caracteres rol VARCHAR(20)
NUMERIC(10,2) Números decimales con precisión exacta precio = 18.50
INTEGER Números enteros (sin decimales) cantidad = 3
BOOLEAN Solo dos valores: verdadero o falso activo = true
TIMESTAMPTZ Fecha y hora con zona horaria created_at = '2024-03-15T10:30:00Z'
JSONB Objetos JSON almacenados en formato binario Datos extras o configuraciones flexibles
Importante: ¿Por qué NUMERIC y no FLOAT para precios?
Los números de punto flotante (FLOAT) tienen errores de precisión por la forma en que la computadora los almacena en binario. Por ejemplo: 0.1 + 0.2 = 0.30000000000000004. Para manejar dinero eso es inaceptable. NUMERIC(10,2) guarda los números con precisión exacta, igual que una calculadora. Siempre usa NUMERIC para precios y cantidades monetarias.
Tip: UUID vs entero autoincremental
Podríamos usar IDs simples como 1, 2, 3... pero los UUID tienen ventajas: son únicos globalmente (puedes generar uno en el frontend sin consultar la BD), no revelan cuántos registros tienes, y son más seguros (nadie puede adivinar el ID de otro usuario poniendo /users/1, /users/2...).

4. Claves Primarias y Claves Foráneas

Clave Primaria (PRIMARY KEY)

Es el identificador único e irrepetible de cada fila en una tabla. Reglas:

  • Ninguna fila puede tener el mismo valor que otra en la clave primaria.
  • No puede ser NULL (vacío).
  • En Aroma usamos UUID generado automáticamente con gen_random_uuid().

Clave Foránea (FOREIGN KEY)

Es una columna que apunta a la clave primaria de otra tabla. Esto es lo que crea las relaciones entre tablas y da nombre a las bases de datos "relacionales".

TABLA: categories          TABLA: products
┌─────┬──────────┐        ┌─────┬─────────────────────┬─────────────┐
│ id  │ nombre   │◄───┐   │ id  │ nombre              │ category_id │
├─────┼──────────┤    │   ├─────┼─────────────────────┼─────────────┤
│ 001 │ Cafés    │    └───│ A1  │ Café Latte          │ 001         │
│ 002 │ Jugos    │        │ A2  │ Café Mocha          │ 001         │
│ 003 │ Pasteles │        │ B1  │ Jugo de Naranja     │ 002         │
└─────┴──────────┘        │ C1  │ Pastel de Chocolate │ 003         │
                          └─────┴─────────────────────┴─────────────┘

products.category_id apunta a categories.id. Esta es una relación muchos-a-uno: muchos productos pueden pertenecer a una misma categoría.

Tip: La integridad referencial
Gracias a la clave foránea, la base de datos evita que pongas un category_id que no existe en la tabla categories. Si intentas insertar un producto con category_id = '999' pero esa categoría no existe, la BD rechaza la operación. Esto se llama integridad referencial y previene datos corruptos.

5. SQL Básico — El lenguaje de las bases de datos

SQL (Structured Query Language) es el idioma estándar para comunicarse con bases de datos relacionales. Tiene muy pocas palabras clave fundamentales y una vez que las aprendes puedes trabajar con cualquier base de datos relacional del mundo.

Las 4 operaciones básicas se llaman CRUD: Create (INSERT), Read (SELECT), Update (UPDATE), Delete (DELETE).

SELECT — Leer datos

Es la operación más usada. Con SELECT le preguntamos a la base de datos que nos devuelva datos.

-- El doble guion -- es un comentario en SQL

-- Seleccionar TODOS los campos de TODOS los productos
SELECT * FROM products;

-- Solo algunos campos (más eficiente que el *)
SELECT nombre, precio FROM products;

-- Con condición WHERE (filtrar)
SELECT * FROM products WHERE activo = true;
SELECT * FROM products WHERE precio < 20;
SELECT * FROM products WHERE nombre = 'Café Latte';

-- Ordenar resultados
SELECT * FROM products ORDER BY precio ASC;       -- Ascendente (menor primero)
SELECT * FROM products ORDER BY created_at DESC;  -- Descendente (más reciente primero)

-- Limitar cantidad de resultados (paginación)
SELECT * FROM products LIMIT 10;
SELECT * FROM products LIMIT 10 OFFSET 20;  -- Del registro 21 al 30

-- Contar registros
SELECT COUNT(*) FROM products;
SELECT COUNT(*) FROM products WHERE activo = true;

-- JOIN: unir datos de dos tablas
-- "Dame los productos con el nombre de su categoría"
SELECT
    p.nombre        AS producto,
    p.precio,
    c.nombre        AS categoria
FROM products p
JOIN categories c ON p.category_id = c.id
WHERE p.activo = true
ORDER BY c.nombre, p.nombre;
Tip: Evita SELECT *
En producción es mejor especificar las columnas que necesitas en lugar de usar *. Así la consulta es más rápida (traes menos datos) y más clara (sabes exactamente qué espera tu código).

INSERT — Agregar datos

Usamos INSERT para añadir nuevas filas a una tabla.

-- Agregar un producto nuevo
INSERT INTO products (nombre, descripcion, precio, category_id, activo)
VALUES ('Café Mocha', 'Espresso con chocolate caliente', 20.00, 'uuid-categoria-cafes', true);

-- Agregar múltiples registros de una vez (más eficiente)
INSERT INTO categories (nombre, icono, orden) VALUES
    ('Cafés',       '☕', 1),
    ('Jugos',       '🍊', 2),
    ('Pasteles',    '🎂', 3),
    ('Sándwiches',  '🥪', 4);

UPDATE — Actualizar datos

Modifica filas existentes. Siempre usa WHERE para no actualizar toda la tabla por accidente.

-- Actualizar el precio de un producto específico
UPDATE products
SET precio = 22.00
WHERE id = 'a1b2-c3d4-e5f6-g7h8';

-- Desactivar todos los productos de una categoría
UPDATE products
SET activo = false
WHERE category_id = 'uuid-categoria-pasteles';

-- Actualizar múltiples columnas a la vez
UPDATE products
SET
    precio      = 20.00,
    descripcion = 'Espresso con leche vaporizada y arte latte'
WHERE id = 'a1b2-c3d4-e5f6-g7h8';

DELETE — Eliminar datos

Elimina filas de una tabla. Es la operación más peligrosa: no se puede deshacer.

-- Eliminar un producto por su ID (forma segura)
DELETE FROM products WHERE id = 'a1b2-c3d4-e5f6-g7h8';

-- Eliminar todos los registros inactivos
DELETE FROM products WHERE activo = false;

-- ⚠️  NUNCA hagas esto sin WHERE:
-- DELETE FROM products;
-- ← Eliminaría TODOS los productos sin posibilidad de recuperarlos
Advertencia: Doble verificación antes de DELETE y UPDATE
Antes de ejecutar un UPDATE o DELETE en producción, es buena práctica ejecutar primero un SELECT con el mismo WHERE para confirmar exactamente qué filas vas a afectar. Una vez ejecutado el DELETE, no hay "Ctrl+Z".

6. ¿Qué es Supabase?

Analogía: Supabase es el "todo incluido" de las bases de datos
Imagina que para abrir una cafetería necesitas: contratar personal, comprar equipos, obtener permisos sanitarios, instalar cámaras de seguridad, etc. Supabase es como tomar un local en un centro comercial donde ya todo eso está incluido. Tú solo te enfocas en tu negocio.

Supabase se autodefine como "la alternativa open source a Firebase". Es una plataforma que te proporciona, de manera gratuita para proyectos pequeños:

  • Base de datos PostgreSQL gestionada: no necesitas instalar ni configurar un servidor de base de datos.
  • API REST automática: sin escribir nada de backend, puedes leer y escribir datos desde tu frontend.
  • Autenticación de usuarios: login/registro con email, Google, GitHub, etc. Sin gestionar contraseñas manualmente.
  • Storage de archivos: guardar imágenes, documentos, etc.
  • Real-time: recibir actualizaciones instantáneas cuando cambian los datos (ideal para el panel del cajero).
  • Dashboard visual: interfaz web para administrar tu BD sin escribir código.

Cómo usa Aroma Supabase

  • El frontend se conecta a Supabase directamente para leer el menú (productos, categorías).
  • El sistema de autenticación (login/registro) usa Supabase Auth completamente.
  • El backend Node.js se conecta a Supabase para operaciones complejas como crear pedidos, calcular totales y actualizar estados.

8. El esquema de base de datos de Aroma

El archivo database/schema.sql contiene todas las instrucciones para crear las tablas del proyecto. Vamos a analizarlo tabla por tabla.

Tabla: profiles (perfiles de usuario)

Almacena información adicional de los usuarios. El usuario base lo crea Supabase Auth; nosotros guardamos aquí los datos extra como nombre completo, teléfono y rol.

CREATE TABLE profiles (
    id          UUID PRIMARY KEY,
    -- El id viene directamente de auth.users de Supabase
    -- (Supabase Auth lo crea automáticamente al registrarse)

    full_name   TEXT,
    phone       TEXT,

    role        VARCHAR(20) NOT NULL DEFAULT 'client'
                CHECK (role IN ('admin', 'cashier', 'client')),
    -- CHECK valida que solo se acepten esos 3 valores exactos
    -- Si intentas poner role = 'superadmin', la BD lo rechaza

    active      BOOLEAN NOT NULL DEFAULT true,
    created_at  TIMESTAMPTZ DEFAULT NOW()

    -- FOREIGN KEY implícita: este perfil pertenece a un usuario de auth.users
    -- REFERENCES auth.users(id) ON DELETE CASCADE
    -- "Si se elimina el usuario de auth, también se elimina su perfil"
);

Tabla: categories (categorías del menú)

CREATE TABLE categories (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    -- gen_random_uuid() genera un UUID automáticamente al insertar

    nombre      TEXT NOT NULL,       -- NOT NULL = obligatorio, no puede estar vacío
    descripcion TEXT,                -- Opcional
    icono       TEXT,                -- Emoji o nombre de ícono
    orden       INTEGER DEFAULT 0,   -- Para ordenar el menú como queramos
    activo      BOOLEAN DEFAULT true,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

Tabla: products (productos del menú)

CREATE TABLE products (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    nombre      TEXT NOT NULL,
    descripcion TEXT,

    precio      NUMERIC(10,2) NOT NULL CHECK (precio > 0),
    -- NUMERIC(10,2): hasta 10 dígitos en total, 2 decimales
    -- CHECK (precio > 0): no se puede guardar un precio negativo o cero

    costo       NUMERIC(10,2),       -- Costo interno (para calcular margen)

    category_id UUID REFERENCES categories(id),
    -- REFERENCES categories(id): clave foránea hacia la tabla categories

    imagen_url  TEXT,
    activo      BOOLEAN DEFAULT true,
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

Tabla: orders (pedidos)

CREATE TABLE orders (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- Código legible: ARO-20240315-1234 (generado automáticamente por trigger)
    code            TEXT UNIQUE,

    -- Estado del pedido (flujo completo)
    status          VARCHAR(30) NOT NULL DEFAULT 'pending'
                    CHECK (status IN (
                        'pending',      -- Recibido, sin confirmar
                        'confirmed',    -- Confirmado por el cajero
                        'preparing',    -- En preparación
                        'ready',        -- Listo para entregar
                        'delivered',    -- Entregado al cliente
                        'cancelled'     -- Cancelado
                    )),

    -- Datos del cliente (en el momento del pedido)
    user_id             UUID REFERENCES auth.users(id),
    customer_name       TEXT NOT NULL,
    customer_phone      TEXT NOT NULL,
    delivery_address    TEXT NOT NULL,

    -- Pago
    payment_method  VARCHAR(20) CHECK (payment_method IN ('qr', 'cash', 'card')),
    total           NUMERIC(10,2) NOT NULL,

    notes           TEXT,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

Tabla: order_items (detalle de los pedidos)

CREATE TABLE order_items (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    order_id    UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
    -- ON DELETE CASCADE: si se elimina el pedido, se eliminan sus items también

    product_id  UUID REFERENCES products(id),
    -- Referencia al producto (puede quedar NULL si el producto se elimina)

    -- *** SNAPSHOT DE PRECIO ***
    -- Guardamos el nombre y precio AL MOMENTO del pedido.
    -- Si el precio del café cambia mañana, este pedido conserva el precio original.
    -- Esto es FUNDAMENTAL en sistemas de ventas reales.
    product_name    TEXT NOT NULL,
    unit_price      NUMERIC(10,2) NOT NULL,

    quantity        INTEGER NOT NULL CHECK (quantity > 0),
    subtotal        NUMERIC(10,2) NOT NULL,  -- unit_price * quantity

    created_at      TIMESTAMPTZ DEFAULT NOW()
);
Concepto clave: SNAPSHOT de precios
¿Por qué guardamos product_name y unit_price en order_items si ya están en la tabla products? Porque los precios cambian con el tiempo. Si un Café Latte costaba Bs. 18 en marzo y lo subimos a Bs. 22 en junio, los pedidos de marzo deben seguir mostrando Bs. 18. Al guardar el precio en el momento exacto de la venta, garantizamos la integridad histórica de todos los registros.

Trigger para el código de pedido

Un trigger es código SQL que se ejecuta automáticamente cuando ocurre un evento en la base de datos (INSERT, UPDATE o DELETE). No lo llamas tú: la BD lo llama sola.

-- Función que genera el código legible del pedido
CREATE OR REPLACE FUNCTION generate_order_code()
RETURNS TRIGGER AS $$
BEGIN
    -- Genera: ARO-20240315-1234
    -- ARO = prefijo de Aroma
    -- 20240315 = fecha en formato YYYYMMDD
    -- 1234 = número aleatorio de 4 dígitos
    NEW.code := 'ARO-'
        || TO_CHAR(NOW(), 'YYYYMMDD')
        || '-'
        || LPAD(FLOOR(RANDOM() * 9000 + 1000)::TEXT, 4, '0');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Trigger: llamar a la función ANTES de cada INSERT en orders
-- Solo cuando el code no viene especificado (IS NULL)
CREATE TRIGGER set_order_code
    BEFORE INSERT ON orders
    FOR EACH ROW
    WHEN (NEW.code IS NULL)
    EXECUTE FUNCTION generate_order_code();
Analogía: El trigger es como un empleado automático
Imagina que cada vez que llega un pedido nuevo, hay un empleado cuya única tarea es estampar un código único en ese pedido. No necesitas recordar llamarlo; está siempre ahí, esperando. El trigger hace exactamente eso: ejecuta código automáticamente cada vez que se inserta un pedido.

Con el proyecto creado, es momento de crear todas las tablas. Supabase incluye un editor SQL directamente en el dashboard.

1
En el panel izquierdo de Supabase, busca el ícono de "SQL Editor" (parece un terminal o pantalla con código).
2
Haz clic en "New query" para abrir un editor en blanco.
3
Abre el archivo database/schema.sql del proyecto Aroma, copia todo su contenido y pégalo en el editor de Supabase.
4
Haz clic en el botón "Run" (o presiona Ctrl+Enter). Si todo está correcto, verás el mensaje "Success. No rows returned".
5
Ve al "Table Editor" (ícono de tabla en el menú izquierdo) y verifica que se crearon las tablas: profiles, categories, products, orders, order_items.
6
Ahora inserta los datos iniciales con la siguiente consulta en el SQL Editor: 
-- ===== DATOS INICIALES DE PRUEBA =====

-- 1. Insertar categorías del menú
INSERT INTO categories (nombre, icono, orden) VALUES
    ('Cafés',       '☕', 1),
    ('Jugos',       '🍊', 2),
    ('Pasteles',    '🎂', 3),
    ('Sándwiches',  '🥪', 4);

-- 2. Insertar productos de prueba usando subconsulta para obtener el category_id
-- Cafés
INSERT INTO products (nombre, descripcion, precio, costo, category_id, activo)
SELECT
    'Café Americano',
    'Espresso suavizado con agua caliente',
    12.00, 3.00, id, true
FROM categories WHERE nombre = 'Cafés';

INSERT INTO products (nombre, descripcion, precio, costo, category_id, activo)
SELECT
    'Café Latte',
    'Espresso con leche vaporizada cremosa',
    18.50, 5.00, id, true
FROM categories WHERE nombre = 'Cafés';

INSERT INTO products (nombre, descripcion, precio, costo, category_id, activo)
SELECT
    'Café Mocha',
    'Espresso con chocolate y crema batida',
    20.00, 6.00, id, true
FROM categories WHERE nombre = 'Cafés';

-- Jugos
INSERT INTO products (nombre, descripcion, precio, costo, category_id, activo)
SELECT
    'Jugo de Naranja Natural',
    'Naranja exprimida al momento, sin azúcar añadida',
    15.00, 4.00, id, true
FROM categories WHERE nombre = 'Jugos';

-- Pasteles
INSERT INTO products (nombre, descripcion, precio, costo, category_id, activo)
SELECT
    'Pastel de Chocolate',
    'Bizcocho húmedo de chocolate con ganache',
    25.00, 8.00, id, true
FROM categories WHERE nombre = 'Pasteles';

-- Verificar que todo se insertó correctamente
SELECT p.nombre, p.precio, c.nombre AS categoria
FROM products p
JOIN categories c ON p.category_id = c.id
ORDER BY c.orden, p.nombre;
Tip: Usa subconsultas para referencias
En lugar de copiar y pegar el UUID de cada categoría manualmente (que sería tedioso y propenso a errores), usamos SELECT id FROM categories WHERE nombre = 'Cafés' dentro del INSERT. PostgreSQL lo resuelve automáticamente. Mucho más limpio.

  • Una base de datos es un sistema para guardar, buscar, actualizar y eliminar información de manera eficiente. Aroma usa PostgreSQL (relacional).
  • Las bases relacionales organizan datos en tablas (columnas + filas), conectadas por claves foráneas.
  • Los tipos de datos más usados son UUID, TEXT, NUMERIC, INTEGER, BOOLEAN y TIMESTAMPTZ. Para precios siempre NUMERIC, nunca FLOAT.
  • SQL es el lenguaje para operar la BD: SELECT (leer), INSERT (crear), UPDATE (modificar), DELETE (eliminar).
  • Supabase es la plataforma que gestiona nuestra BD PostgreSQL y nos da auth, API y dashboard gratis.
  • El esquema de Aroma tiene 5 tablas: profiles, categories, products, orders, order_items.
  • El concepto de SNAPSHOT de precios en order_items garantiza la integridad histórica de los pedidos.
  • Los triggers son código SQL que se ejecuta automáticamente ante eventos en la BD, como generar el código de pedido.
