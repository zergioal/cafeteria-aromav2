Guía 15 – Despliegue: Publicando Aroma en Internet

Módulo 6: Despliegue | De localhost a producción real

Nivel: Avanzado Tiempo estimado: 4 horas Prerequisito: Guías 11-14

Objetivos de esta guía

  • Entender la diferencia entre desarrollo y producción
  • Subir el código a GitHub con buenas prácticas
  • Desplegar el backend en Render (plataforma gratuita)
  • Desplegar el frontend en Netlify (plataforma gratuita)
  • Configurar las variables de entorno en producción
  • Verificar que todo funciona desde internet
  • Conocer el flujo CI/CD básico para actualizar el proyecto

1. ¿Qué es el despliegue (deployment)?

Analogía del restaurante: Hasta ahora, la Cafetería Aroma solo "existía" en tu cocina de casa (localhost). El despliegue es como abrir el local real: conectar la electricidad, poner el letrero en la puerta, y habilitar las mesas para que cualquier persona pueda venir a comer, desde cualquier ciudad del mundo.

Desarrollo vs Producción

Aspecto Desarrollo (localhost) Producción (internet)
Acceso Solo tu computadora Cualquier persona en el mundo
URL http://localhost:3000 https://aroma-backend.onrender.com
Errores visibles Sí, con detalles (para depurar) No, mensajes genéricos (seguridad)
Nodemon Sí (reinicia al guardar) No, se usa node directamente
.env Archivo local en el proyecto Variables configuradas en el servidor
HTTPS No necesario Obligatorio (navegadores lo exigen)

La arquitectura de Aroma en producción

[Usuarios en internet] | v [Netlify — Frontend] ←→ [Supabase — Auth + BD] | | v | [Render — Backend Node.js] ←----------+ (API REST)

Cada capa tiene una responsabilidad clara y su propia plataforma de hosting:

  • Netlify: Sirve los archivos estáticos del frontend (HTML, CSS, JS). Gratuito, muy rápido.
  • Render: Ejecuta el servidor Node.js/Express. Plan gratuito disponible.
  • Supabase: Aloja la base de datos PostgreSQL y el sistema de autenticación. Ya lo teníamos configurado.

2. Git y GitHub — Preparar el código

Antes de desplegar, el código debe estar en GitHub. Git es el sistema de control de versiones que guarda el historial de cambios; GitHub es la plataforma web donde se aloja ese repositorio.

Verificar el .gitignore ANTES de subir a GitHub: El archivo .gitignore le dice a Git qué archivos ignorar. Las credenciales, los node_modules y los archivos de caché nunca deben subirse. 
# .gitignore (en la raíz del proyecto)
# Variables de entorno con credenciales
.env
.env.local
.env.production

# Dependencias de Node (se instalan con npm install)
node_modules/

# Logs
*.log
npm-debug.log*

# Archivos del sistema operativo
.DS_Store
Thumbs.db

# Archivos de VS Code
.vscode/settings.json
# Comandos para subir el código a GitHub:

# 1. Si el repositorio no existe todavía, inicializarlo
git init

# 2. Verificar qué archivos se van a incluir (¡no debe aparecer .env!)
git status

# 3. Revisar que .gitignore está funcionando
git check-ignore -v .env
# Debe mostrar: .gitignore:1:.env	.env

# 4. Agregar todos los archivos al staging
git add .

# 5. Crear el primer commit
git commit -m "feat: Cafetería Aroma - sistema fullstack completo"

# 6. Ir a github.com → New Repository → nombre: cafeteria-aroma
#    (NO inicializar con README ni .gitignore, ya los tenemos)

# 7. Conectar con GitHub y subir
git remote add origin https://github.com/tu-usuario/cafeteria-aroma.git
git branch -M main
git push -u origin main
Tip: Después de crear el repositorio en GitHub, la propia plataforma te muestra exactamente estos comandos en pantalla. Solo tienes que copiarlos y ejecutarlos en tu terminal.

3. Desplegar el Backend en Render

Render es una plataforma de hosting que puede ejecutar servidores Node.js de forma gratuita. La conexión con GitHub es directa: cada vez que subas cambios, Render redespliega automáticamente.

1
Ir a render.com y registrarse con la cuenta de GitHub (más fácil así).
2
En el dashboard, clic en "New +" → "Web Service".
3
Seleccionar "Build and deploy from a Git repository" → conectar GitHub y buscar el repositorio cafeteria-aroma.
4
Configurar el servicio: 
Name:              aroma-backend
Root Directory:    backend
Environment:       Node
Build Command:     npm install
Start Command:     npm start
5
Ir a la sección "Environment" y agregar las variables: 
SUPABASE_URL            = https://tu-proyecto.supabase.co
SUPABASE_SERVICE_ROLE_KEY = eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
FRONTEND_URL            = *
(Ponemos * en FRONTEND_URL temporalmente; la actualizaremos cuando tengamos la URL de Netlify.)
6
Clic en "Create Web Service". Render comenzará a construir el servidor (3-5 minutos).
7
Cuando el estado sea "Live", probar el health check en el navegador: https://aroma-backend.onrender.com/health. Debe responder { "status": "OK" }.
Plan gratuito de Render — Cold Start: En el plan gratuito, Render "duerme" el servidor después de 15 minutos sin peticiones. La primera petición después de que durmió puede tardar 30-60 segundos en responder (mientras el servidor "despierta"). Esto es normal en desarrollo/demo. Para proyectos reales en producción, se usa el plan de pago.

4. Desplegar el Frontend en Netlify

Netlify es la plataforma ideal para sitios web estáticos. Detecta automáticamente que no necesitamos un proceso de build y sirve los archivos directamente.

1
Ir a netlify.com y registrarse con GitHub.
2
En el dashboard, clic en "Add new site" → "Import an existing project".
3
Seleccionar GitHub y buscar cafeteria-aroma.
4
Configurar el despliegue: 
Branch to deploy:    main
Base directory:      frontend
Publish directory:   frontend
Build command:       (dejar vacío — no usamos build)
5
Clic en "Deploy site". En aproximadamente 1 minuto obtendrás una URL como: https://amazing-curie-a1b2c3.netlify.app
6
Opcional: En Site settings → General → "Change site name" puedes personalizar el subdominio: aroma-cafeteria.netlify.app

5. Actualizar las URLs de producción

Ahora que tenemos las URLs reales de Render y Netlify, debemos actualizar las configuraciones.

Paso A — Actualizar frontend/js/config.js

// frontend/js/config.js
// Este archivo SÍ se sube a GitHub (no tiene secretos, solo URLs públicas)
const CONFIG = {
    // Tu proyecto de Supabase (igual que en desarrollo)
    SUPABASE_URL: 'https://tu-proyecto.supabase.co',

    // La clave pública de Supabase (ANON KEY — es segura para el frontend)
    SUPABASE_ANON_KEY: 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...',

    // URL del backend en producción (reemplazar con tu URL real de Render)
    API_BASE_URL: 'https://aroma-backend.onrender.com'
};
Importante: El archivo config.js usa la ANON_KEY de Supabase, que es pública por diseño. La SERVICE_ROLE_KEY (la secreta) solo está en las variables de entorno de Render, nunca en el frontend ni en GitHub.

Paso B — Actualizar FRONTEND_URL en Render

1
Ir a Render → tu servicio aroma-backend → pestaña "Environment".
2
Cambiar el valor de FRONTEND_URL de * a la URL real de Netlify: https://aroma-cafeteria.netlify.app
3
Clic en "Save Changes". Render reiniciará el servidor automáticamente.

Paso C — Configurar Supabase Authentication

1
Ir a tu proyecto en supabase.com → Authentication → URL Configuration.
2
En "Redirect URLs", agregar: https://aroma-cafeteria.netlify.app/**
3
También actualizar el Site URL a la URL de Netlify.

Paso D — Subir los cambios

# Agregar los cambios a Git y subir
git add frontend/js/config.js
git commit -m "config: actualizar URLs de producción"
git push origin main

# Netlify y Render detectarán el push automáticamente y redesplegarán

6. El archivo netlify.toml — Configuración de rutas

Cuando el usuario accede directamente a una URL como https://aroma-cafeteria.netlify.app/menu, Netlify buscaría un archivo /menu/index.html que no existe. El resultado sería un error 404. Para evitar esto en nuestra Single Page Application (SPA), necesitamos el archivo netlify.toml:

# frontend/netlify.toml
# Este archivo configura el comportamiento de Netlify

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200

# Opcional: agregar cabeceras de seguridad
[[headers]]
  for = "/*"
  [headers.values]
    X-Frame-Options = "DENY"
    X-Content-Type-Options = "nosniff"
¿Por qué status 200 y no 301? Con status = 200, Netlify devuelve el contenido de index.html pero mantiene la URL original en el navegador. Con un redirect 301, el navegador cambiaría la URL a /index.html, lo que no es lo que queremos. Siempre 200 para SPAs.

7. Verificar el despliegue completo

Una vez que todo está desplegado, usa este checklist para verificar que funciona correctamente desde internet:

  • El backend responde en https://aroma-backend.onrender.com/health
  • El frontend carga en la URL de Netlify
  • Los productos se muestran en el menú (datos de la API)
  • El registro de nuevo usuario funciona
  • El login funciona
  • Se pueden agregar productos al carrito
  • El checkout crea el pedido correctamente
  • El cajero puede ver los pedidos desde su panel
  • El cambio de estados del pedido funciona
  • El admin puede ver el dashboard con estadísticas
  • La creación y edición de productos funciona
  • No hay errores CORS en la consola del navegador

8. Errores comunes en producción

Error Causa probable Solución
Mixed Content El frontend usa HTTPS pero llama a un backend HTTP Render proporciona HTTPS automáticamente. Asegurarse de usar la URL https:// en config.js
CORS en producción FRONTEND_URL en Render tiene el valor antiguo o incorrecto Actualizar la variable en Render con la URL exacta de Netlify
500 Internal Server Error Variables de entorno no configuradas en Render Verificar en Render → Environment que SUPABASE_URL y SUPABASE_SERVICE_ROLE_KEY están correctas
Login no redirige URL de redirección no configurada en Supabase Agregar la URL de Netlify en Supabase → Authentication → Redirect URLs
Página 404 al recargar Falta el archivo netlify.toml Crear el archivo con la regla de redirect a index.html
Primera petición tarda 60s Cold start del plan gratuito de Render Normal en el plan gratuito. Se puede agregar un "ping" periódico o usar el plan de pago
Productos no cargan La URL de la API en config.js es incorrecta Verificar que API_BASE_URL en config.js tiene la URL exacta de Render (sin barra al final)

9. CI/CD básico — El flujo de trabajo en producción

CI/CD (Continuous Integration / Continuous Deployment) es la práctica de automatizar las pruebas y el despliegue. En su forma más básica, con Render y Netlify conectados a GitHub, ya tienes CI/CD: cada vez que subes código, se despliega automáticamente.

# El flujo de trabajo diario para actualizar el proyecto:

# 1. Hacer cambios en el código (agregar una feature, corregir un bug)
# ...editar archivos...

# 2. Revisar qué cambió
git status
git diff

# 3. Agregar los cambios
git add .

# 4. Crear un commit descriptivo
git commit -m "feat: agregar filtro de búsqueda en el menú"

# 5. Subir a GitHub
git push origin main

# ¡Listo! Render y Netlify detectan el push y redesplegan automáticamente
# El proceso tarda ~2-3 minutos
# Puedes ver el progreso en el dashboard de cada plataforma

Buenas prácticas de commit

Los mensajes de commit deben ser descriptivos y seguir una convención. La más usada es Conventional Commits:

Prefijo Uso Ejemplo
feat: Nueva funcionalidad feat: agregar filtro de categorías en el menú
fix: Corrección de bug fix: corregir cálculo de total cuando hay descuento
style: Cambios de CSS/estilo style: mejorar diseño del carrito en mobile
refactor: Reestructuración sin cambiar funcionalidad refactor: extraer lógica del carrito a módulo propio
docs: Documentación docs: actualizar README con instrucciones de instalación
config: Cambios de configuración config: actualizar URLs de producción

Ejercicio Final — Checklist completo de despliegue

Sigue estos pasos en orden para publicar la Cafetería Aroma en internet:

1
Preparar el código: Verificar que .gitignore excluye .env y node_modules. Crear el archivo backend/.env.example con las claves vacías.
2
Subir a GitHub: Crear el repositorio en github.com, conectar y hacer el primer git push. Verificar que .env no aparece en los archivos del repositorio.
3
Desplegar backend en Render: Conectar el repositorio, configurar Root Directory = backend, Start Command = npm start. Agregar las variables de entorno.
4
Probar el backend: Abrir la URL de Render + /health en el navegador. Esperar a que el servidor "despierte" (puede tardar 60 segundos la primera vez).
5
Desplegar frontend en Netlify: Conectar el repositorio, configurar Base directory = frontend. Agregar el archivo netlify.toml.
6
Actualizar config.js: Poner la URL real de Render en API_BASE_URL. Hacer commit y push.
7
Actualizar FRONTEND_URL en Render: Cambiar el valor a la URL real de Netlify para que CORS funcione.
8
Configurar Supabase Auth: Agregar la URL de Netlify en Redirect URLs dentro de Supabase.
9
Prueba de humo (smoke test): Abrir la URL de Netlify desde el celular o una red diferente. Hacer el flujo completo: ver menú → login → agregar al carrito → confirmar pedido.
10
Compartir: Enviar la URL a un amigo o compañero y pedirle que pruebe la aplicación. Los ojos frescos detectan cosas que tú ya no ves.

Resumen de la Guía 15

  • El despliegue hace que la aplicación sea accesible desde internet para cualquier persona.
  • El código debe estar en GitHub antes de desplegarlo; el .env nunca se sube.
  • Render hospeda el backend Node.js y lee las variables de entorno desde su panel.
  • Netlify hospeda el frontend estático y requiere netlify.toml para las rutas de SPA.
  • Después del despliegue hay que actualizar tres lugares: config.js, FRONTEND_URL en Render, y Redirect URLs en Supabase.
  • El cold start del plan gratuito de Render es normal: la primera petición tarda más.
  • Con CI/CD básico, cada git push redespliega automáticamente Render y Netlify.
