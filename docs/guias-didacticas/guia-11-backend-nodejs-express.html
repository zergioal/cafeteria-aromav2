1. ¿Qué es un servidor backend?

Analogía del restaurante: Si el frontend es la sala del restaurante donde están sentados los clientes (los usuarios mirando la web), el backend es la cocina donde realmente se prepara todo. El cliente (el navegador) hace pedidos (requests), el cocinero (el servidor backend) los procesa y devuelve los platos terminados (responses con datos JSON).

Hasta ahora, nuestro frontend de Aroma hablaba directamente con Supabase. Eso funciona para operaciones simples, pero tiene un problema serio: necesitamos manejar lógica de negocio compleja y proteger credenciales sensibles.

¿Por qué necesitamos un backend propio?

  • Seguridad: La clave SERVICE_ROLE_KEY de Supabase da acceso total a la base de datos. Si la ponemos en el frontend, cualquier persona la puede ver inspeccionando el código. El backend la guarda en el servidor, lejos del navegador.
  • Lógica de negocio: Calcular totales de pedidos, verificar stock, aplicar descuentos. Estas reglas deben vivir en el servidor, no en el navegador (donde el usuario podría modificarlas).
  • Autorización: Un cliente no debería poder ver los pedidos de otro cliente. El backend verifica quién eres y qué tienes permitido hacer.
  • Validación: Un pedido no puede tener 0 items, un precio no puede ser negativo. El backend valida estos casos antes de guardar datos.
  • Operaciones complejas: Generar reportes, enviar emails, integrar con sistemas de pago — todo esto vive en el backend.

En la Cafetería Aroma:

Nuestro backend es el encargado de recibir los pedidos del cliente, verificar que los productos existan y estén activos, calcular el total con los precios reales de la base de datos (nunca confiando en lo que manda el navegador), y guardar el pedido. También protege las rutas del cajero y del administrador para que solo los usuarios con el rol correcto puedan acceder.

2. Node.js: JavaScript fuera del navegador

JavaScript nació para vivir en los navegadores web. Pero en 2009, Ryan Dahl creó Node.js: un entorno que permite ejecutar JavaScript en el servidor, fuera del navegador. Node.js usa el mismo motor V8 de Google Chrome que conocemos, pero en lugar de controlar páginas web, controla el sistema operativo, archivos, redes y servidores.

Analogía: Es como si el idioma español (JavaScript) que antes solo se hablaba en España (el navegador), ahora también se pudiera hablar en toda América Latina (el servidor). El mismo idioma, contextos totalmente diferentes.

npm: el gestor de paquetes

Node.js viene con npm (Node Package Manager), que es como una tienda de herramientas para desarrolladores. Con npm podemos instalar librerías que otras personas ya crearon en lugar de escribir todo desde cero.

# Inicializar un proyecto nuevo (crea el archivo package.json)
npm init -y

# Instalar Express (el framework web)
npm install express

# Instalar dependencias de desarrollo (solo para trabajar, no para producción)
npm install --save-dev nodemon

# Instalar varias a la vez
npm install cors morgan dotenv @supabase/supabase-js

# Ejecutar el servidor (según scripts en package.json)
npm start          # Modo producción
npm run dev        # Modo desarrollo (con nodemon)
Tip: nodemon es una herramienta mágica para desarrollo: reinicia el servidor automáticamente cada vez que guardas un archivo. Sin nodemon, tendrías que detener y reiniciar el servidor manualmente después de cada cambio.

El archivo package.json

Este archivo es el "pasaporte" de tu proyecto Node.js. Contiene el nombre, la versión, las dependencias y los scripts para ejecutar el proyecto.

{
  "name": "aroma-backend",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "start": "node src/index.js",
    "dev": "nodemon src/index.js"
  },
  "dependencies": {
    "express": "^4.18.2",
    "cors": "^2.8.5",
    "morgan": "^1.10.0",
    "dotenv": "^16.3.1",
    "@supabase/supabase-js": "^2.38.0"
  },
  "devDependencies": {
    "nodemon": "^3.0.1"
  }
}
Importante: La línea "type": "module" le dice a Node.js que usamos la sintaxis moderna de ES Modules (import/export) en lugar de la antigua CommonJS (require()). En este proyecto usamos la sintaxis moderna.

3. Express.js: el framework web

Express.js es el framework web más popular para Node.js. Sin Express, crear un servidor HTTP en Node puro requiere manejar decenas de detalles tediosos: parsear URLs, manejar distintos métodos HTTP, leer el body de las peticiones, gestionar cabeceras... Express lo simplifica todo con una API elegante y concisa.

Tu primer servidor Express

// El servidor Express más básico posible
import express from 'express';

const app = express();
const PORT = 3000;

// Ruta GET a "/"
app.get('/', (req, res) => {
    res.json({ mensaje: 'Cafetería Aroma API funcionando ✅' });
});

// Iniciar el servidor
app.listen(PORT, () => {
    console.log(`Servidor corriendo en http://localhost:${PORT}`);
});

Con solo 10 líneas ya tienes un servidor web funcionando. Cuando abres http://localhost:3000/ en el navegador, Express recibe la petición GET, ejecuta la función (el "route handler"), y devuelve el JSON al navegador.

Los objetos req y res

Cada ruta recibe dos objetos fundamentales:

  • req (request): Contiene todo sobre la petición entrante: la URL, el método HTTP, los headers, el body, los parámetros de la URL, la IP del cliente, etc.
  • res (response): Lo usamos para enviar la respuesta: res.json() para JSON, res.send() para texto, res.status(404).json() para errores con código de estado.

4. HTTP Methods y el diseño REST

HTTP define varios "métodos" o "verbos" que indican qué tipo de operación se quiere realizar. En una API REST, cada método tiene un significado específico:

Método Acción Ejemplo en Cafetería Aroma Código de éxito
GET Obtener / leer datos GET /api/products — listar todos los productos 200 OK
POST Crear un nuevo recurso POST /api/orders — crear un pedido nuevo 201 Created
PUT Reemplazar un recurso completo PUT /api/products/:id — actualizar todos los datos de un producto 200 OK
PATCH Modificar parcialmente PATCH /api/orders/:id/status — solo cambiar el estado 200 OK
DELETE Eliminar un recurso DELETE /api/products/:id — eliminar un producto 200 OK

Convenciones REST

REST (Representational State Transfer) es un estilo de diseño para APIs. Sus reglas más importantes son:

  • Usa sustantivos, no verbos: La ruta debe ser /api/products, no /api/getProducts. El verbo lo aporta el método HTTP.
  • Usa plural para colecciones: /api/products (lista), /api/products/5 (uno específico).
  • Anida recursos relacionados: /api/orders/5/items para los items del pedido 5.
  • Devuelve códigos de estado HTTP correctos: 200 para éxito, 201 para creación, 400 para datos inválidos, 401 para no autenticado, 403 para no autorizado, 404 para no encontrado, 500 para error del servidor.

5. El servidor real de Aroma (index.js)

Ahora veamos el servidor completo de la Cafetería Aroma. Cada parte cumple una función específica:

// backend/src/index.js
import express from 'express';
import cors from 'cors';
import morgan from 'morgan';
import { config } from 'dotenv';

// Cargar variables de entorno del archivo .env
config();

import publicRoutes from './routes/public.js';
import clientRoutes from './routes/client.js';
import cashierRoutes from './routes/cashier.js';
import adminRoutes from './routes/admin.js';

const app = express();
const PORT = process.env.PORT || 3000;

// ===== MIDDLEWARES =====

// CORS: permitir peticiones desde el frontend
app.use(cors({
    origin: process.env.FRONTEND_URL || 'http://localhost:5500',
    credentials: true
}));

// Parsear JSON del body de las peticiones
app.use(express.json());

// Logging de peticiones en consola: GET /api/products 200 15ms
app.use(morgan('dev'));

// ===== RUTAS =====
app.use('/api', publicRoutes);           // Sin autenticación requerida
app.use('/api', clientRoutes);           // Con autenticación (clientes)
app.use('/api/cashier', cashierRoutes);  // Solo cajeros y admin
app.use('/api/admin', adminRoutes);      // Solo administradores

// Health check: para verificar que el servidor está vivo
app.get('/health', (req, res) => {
    res.json({ status: 'OK', timestamp: new Date().toISOString() });
});

// ===== MANEJO DE ERRORES =====
// Este middleware especial recibe 4 parámetros (err, req, res, next)
app.use((err, req, res, next) => {
    console.error(err.stack);
    res.status(err.status || 500).json({
        error: err.message || 'Error interno del servidor'
    });
});

// 404 para rutas que no existen
app.use((req, res) => {
    res.status(404).json({ error: 'Ruta no encontrada' });
});

app.listen(PORT, () => {
    console.log(`Servidor Aroma corriendo en puerto ${PORT}`);
});

Desglose de cada parte

¿Qué es un middleware?

Un middleware es una función que se ejecuta entre que llega la petición y que se envía la respuesta. Son como "filtros" o "capas" que procesan la petición en orden. Cada middleware puede:

  • Modificar req o res
  • Terminar la petición enviando una respuesta
  • Llamar a next() para pasar al siguiente middleware

CORS — Cross-Origin Resource Sharing

Por defecto, los navegadores bloquean las peticiones JavaScript hacia un dominio diferente. Como el frontend está en localhost:5500 y el backend en localhost:3000, son "orígenes" distintos. El middleware CORS agrega cabeceras especiales a las respuestas que le dicen al navegador "está bien, este dominio tiene permiso".

express.json()

Cuando el frontend envía datos (por ejemplo, el formulario de un pedido), los manda en el body de la petición como texto JSON. express.json() lee ese texto y lo convierte automáticamente en un objeto JavaScript accesible como req.body. Sin este middleware, req.body estaría vacío.

Morgan — el logger

Morgan imprime en la consola del servidor una línea por cada petición recibida, con el método, la ruta, el código de respuesta y el tiempo. Es invaluable para depurar durante el desarrollo:

GET /api/products 200 45ms
POST /api/orders 201 120ms
GET /api/products/99 404 8ms

El manejador de errores global

Express tiene una forma especial de manejar errores: un middleware con 4 parámetros (err, req, res, next). Cuando cualquier parte del código llama a next(error) o lanza una excepción, Express la captura y la envía a este manejador. Es el "plan B" que evita que el servidor se caiga.

6. Middlewares y el ciclo Request-Response

Entender el flujo de una petición es fundamental. Imagina el camino que recorre una petición GET /api/products:

1
Llega la petición al servidor desde el navegador.
2
CORS middleware verifica el origen y agrega los headers de permiso.
3
express.json() lee el body si hay uno y lo pone en req.body.
4
Morgan registra la petición en la consola.
5
El router busca la ruta que coincide con GET /api/products.
6
El controlador consulta Supabase y devuelve los datos con res.json().
// Un middleware personalizado — puede ser cualquier función
function logearTiempo(req, res, next) {
    console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
    next(); // CRÍTICO: siempre llamar next() para continuar el flujo
            // Si no llamas next(), la petición queda "colgada" para siempre
}

// Aplicar a todas las rutas
app.use(logearTiempo);

// Aplicar solo a rutas específicas
app.use('/api/admin', logearTiempo);
Cuidado: Si en un middleware olvidas llamar next() y tampoco envías una respuesta con res.json() o similar, la petición quedará "colgada" indefinidamente hasta que el cliente la cancele por timeout. Es uno de los errores más frecuentes al aprender Express.

7. Rutas y Controladores

A medida que la aplicación crece, poner todas las rutas en index.js se vuelve inmanejable. Por eso organizamos el código en dos capas:

  • Rutas (routes/): Definen qué URL responde a qué función. Solo organizan, no tienen lógica.
  • Controladores (controllers/): Tienen la lógica real: consultar la BD, validar datos, formatear la respuesta.
// backend/src/routes/public.js
// Este archivo solo define las rutas públicas (sin autenticación)
import { Router } from 'express';
import { getProducts, getProductById } from '../controllers/productsController.js';
import { getCategories } from '../controllers/categoriesController.js';

const router = Router();

// Productos disponibles al público
router.get('/products', getProducts);
router.get('/products/:id', getProductById);

// Categorías
router.get('/categories', getCategories);

export default router;
// backend/src/controllers/productsController.js
import supabase from '../services/supabase.js';

// GET /api/products
// Soporta filtros: /api/products?category=001&search=latte
export async function getProducts(req, res) {
    try {
        // Leer parámetros de la query string
        const { category, search } = req.query;

        // Construir la consulta base
        let query = supabase
            .from('products')
            .select(`
                id,
                nombre,
                descripcion,
                precio,
                imagen_url,
                category:categories(id, nombre)
            `)
            .eq('activo', true)
            .order('nombre');

        // Agregar filtro de categoría si se especificó
        if (category) {
            query = query.eq('category_id', category);
        }

        // Agregar búsqueda por nombre si se especificó
        if (search) {
            query = query.ilike('nombre', `%${search}%`);
        }

        const { data: products, error } = await query;

        if (error) throw error;

        res.json({ products });

    } catch (error) {
        console.error('Error obteniendo productos:', error);
        res.status(500).json({ error: 'Error al obtener productos' });
    }
}

// GET /api/products/:id
export async function getProductById(req, res) {
    try {
        const { id } = req.params; // Extraer el :id de la URL

        const { data: product, error } = await supabase
            .from('products')
            .select('*, category:categories(id, nombre)')
            .eq('id', id)
            .eq('activo', true)
            .single(); // Esperamos exactamente un resultado

        if (error || !product) {
            return res.status(404).json({ error: 'Producto no encontrado' });
        }

        res.json({ product });

    } catch (error) {
        res.status(500).json({ error: 'Error al obtener el producto' });
    }
}
Tip — req.params vs req.query vs req.body:
  • req.params: los segmentos dinámicos de la URL. En /products/:id, req.params.id es el valor real (ej: "42").
  • req.query: los parámetros después del ?. En /products?category=cafes, req.query.category es "cafes".
  • req.body: los datos enviados en el cuerpo de la petición (en POST/PUT), como el JSON de un formulario.

8. Variables de entorno (.env)

Las credenciales (contraseñas, claves API, URLs secretas) nunca deben estar escritas directamente en el código. Si subes el código a GitHub con una clave de Supabase, cualquiera puede verla y usar tu base de datos. La solución son las variables de entorno.

# backend/.env
# NUNCA subir este archivo a Git — tiene tus credenciales reales
PORT=3000
SUPABASE_URL=https://tu-proyecto.supabase.co
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
FRONTEND_URL=http://localhost:5500
# backend/.env.example
# Este archivo SÍ va a Git — es la "plantilla" para otros desarrolladores
PORT=3000
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
FRONTEND_URL=
# backend/.gitignore
# Asegúrate de que .env esté aquí
.env
node_modules/
*.log
// Usar las variables de entorno en el código
// dotenv carga el .env cuando llamamos config() al inicio de index.js
import { config } from 'dotenv';
config(); // Esto debe ejecutarse antes de usar process.env

const puerto = process.env.PORT;
const supabaseUrl = process.env.SUPABASE_URL;
const servicioKey = process.env.SUPABASE_SERVICE_ROLE_KEY;

// Usar en la inicialización de Supabase
const supabase = createClient(supabaseUrl, servicioKey);
Importante: La SERVICE_ROLE_KEY de Supabase tiene permisos de superusuario: puede leer, escribir y borrar cualquier dato, saltándose todas las reglas de seguridad (RLS). Por eso solo debe existir en el backend, nunca en el frontend. El frontend usa la ANON_KEY que tiene permisos limitados.

9. Probando la API con Thunder Client

Thunder Client es una extensión de VS Code que funciona como un cliente HTTP: te permite hacer peticiones a tu API y ver las respuestas, sin necesidad de escribir código de prueba. Es ideal para probar los endpoints mientras los desarrollas.

Instalación y primeros pasos

1
Abrir VS Code → ir a Extensions (Ctrl+Shift+X) → buscar "Thunder Client" → instalar.
2
Aparecerá un ícono de rayo en el panel lateral izquierdo. Hacer clic en él.
3
Clic en "New Request" para crear una petición nueva.
4
Seleccionar el método GET y escribir la URL: http://localhost:3000/api/products
5
Clic en "Send" y ver la respuesta JSON en el panel inferior.

Peticiones de prueba básicas

Qué probar Método URL
Verificar que el servidor vive GET http://localhost:3000/health
Listar todos los productos GET http://localhost:3000/api/products
Filtrar por categoría GET http://localhost:3000/api/products?category=001
Buscar por nombre GET http://localhost:3000/api/products?search=latte
Obtener un producto GET http://localhost:3000/api/products/1
Listar categorías GET http://localhost:3000/api/categories
Tip: Guarda las peticiones en colecciones dentro de Thunder Client (como carpetas). Así no tienes que reescribir las URLs cada vez. Crea una colección llamada "Aroma API" y organiza las peticiones por sección: Públicas, Cliente, Cajero, Admin.

Ejercicio Final — Construir y probar el endpoint de productos

Es momento de levantar el servidor y ver todo funcionando en vivo.

1
Abrir la terminal en la carpeta backend/ e instalar las dependencias: npm install
2
Crear el archivo backend/.env copiando .env.example y llenando con tus credenciales reales de Supabase.
3
Iniciar el servidor en modo desarrollo: npm run dev. Deberías ver en la consola: "Servidor Aroma corriendo en puerto 3000".
4
Abrir Thunder Client y probar: GET http://localhost:3000/health. Debe responder { "status": "OK" }.
5
Probar GET http://localhost:3000/api/products. Deberías ver la lista de productos de tu base de datos.
6
Probar el filtro: GET http://localhost:3000/api/products?category=001. Solo deben aparecer los productos de esa categoría.
7
Modifica el nombre de un producto en el controlador (agrega un texto de prueba) y guarda el archivo. Observa cómo nodemon reinicia el servidor automáticamente.

Resumen de la Guía 11

  • El backend es el servidor que procesa la lógica, protege credenciales y controla el acceso.
  • Node.js permite ejecutar JavaScript en el servidor usando el motor V8.
  • npm gestiona las librerías y dependencias del proyecto.
  • Express.js simplifica la creación de servidores HTTP y APIs.
  • Los métodos HTTP (GET, POST, PUT, PATCH, DELETE) indican el tipo de operación.
  • Los middlewares son funciones que procesan la petición en secuencia; siempre deben llamar next().
  • Las variables de entorno (.env) protegen las credenciales del código fuente.
  • Separar rutas y controladores mantiene el código organizado y mantenible.
  • Thunder Client permite probar los endpoints de la API sin escribir código.
