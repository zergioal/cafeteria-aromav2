1. ¿Qué vamos a construir?

☕ Proyecto: Cafetería Aroma

Vamos a construir un sistema web completo y real para una cafetería. No es un ejercicio de práctica sin sentido: es una aplicación que podría usarse en un negocio real desde el primer día.

Cuando terminemos el curso, habremos creado un sistema con cinco partes principales:

1
Menú Online — Los clientes podrán ver todos los productos de la cafetería organizados por categorías (cafés, tés, postres, etc.) con fotos, descripciones y precios. Todo actualizado en tiempo real.
2
Sistema de Pedidos con Carrito — El cliente hace clic en los productos que quiere, los agrega a un carrito de compras (igual que Amazon o MercadoLibre), puede modificar cantidades y ver el total.
3
Pago por QR o en Caja — Al finalizar el pedido, el cliente puede elegir pagar escaneando un código QR desde su celular, o pagar directamente en la caja del local.
4
Panel del Cajero — El empleado de la caja tiene su propia pantalla donde ve todos los pedidos que llegan en tiempo real, puede marcarlos como "en preparación", "listo" o "entregado".
5
Panel de Administrador — El dueño del negocio puede agregar o eliminar productos, ver reportes de ventas, gestionar usuarios y configurar el sistema.

2. ¿Cómo funciona Internet? (Teoría fundamental)

Antes de escribir una sola línea de código, necesitas entender el "mundo" en el que va a vivir tu aplicación. No te preocupes: lo explicaremos con analogías muy simples.

Cliente y Servidor

Analogía del restaurante:
Imagina un restaurante físico. Tú (el cliente) te sientas en una mesa, miras la carta y le haces un pedido al mozo. El mozo lleva tu pedido a la cocina (el servidor). La cocina prepara tu plato y el mozo te lo trae. Tú no necesitas saber cómo se cocina; solo pides y recibes.

En internet funciona exactamente igual:
  • Cliente = tu navegador (Chrome, Firefox). Es quien hace las peticiones.
  • Servidor = una computadora en internet que recibe pedidos y responde con información.
  • Mozo = el protocolo HTTP, el "mensajero" entre cliente y servidor.

¿Qué es HTTP?

HTTP significa HyperText Transfer Protocol (Protocolo de Transferencia de HiperTexto). Es el idioma que usan el cliente y el servidor para comunicarse.

Analogía del idioma:
Si tú hablas español y el mozo habla español, se entienden perfectamente. HTTP es el "español" de internet: un conjunto de reglas que tanto el navegador como el servidor conocen, para poder entenderse.

Cuando escribes una dirección en tu navegador, este le dice al servidor algo como:

GET /menu HTTP/1.1
Host: www.cafeteria-aroma.com

Que en español significa: "Hola servidor de cafeteria-aroma.com, ¿me puedes dar la página del menú?"

¿Qué es una URL?

Una URL (Uniform Resource Locator) es la dirección exacta de un recurso en internet. Analicemos una:

https://www.cafeteria-aroma.com/menu?categoria=cafes#americano

   ↑      ↑          ↑           ↑          ↑          ↑
protocolo subdominio dominio    ruta     parámetro   ancla/hash
  • https:// — El protocolo (HTTP seguro, con cifrado)
  • www. — Subdominio (opcional, pero muy común)
  • cafeteria-aroma.com — El dominio: el "nombre" del servidor en internet
  • /menu — La ruta: qué página o recurso queremos
  • ?categoria=cafes — Parámetros de búsqueda (filtros, opciones)
  • #americano — Ancla: ir directo a una sección dentro de la página

Frontend vs. Backend

Analogía del restaurante (continuando):
  • Frontend = La sala del restaurante: las mesas, la decoración, el menú impreso, lo que el cliente VE y TOCA. Es todo lo que muestra el navegador: el HTML, CSS y JavaScript que se ejecuta en tu computadora.
  • Backend = La cocina: el cliente no la ve, pero es donde se prepara TODO. El servidor recibe pedidos, procesa la lógica, consulta la base de datos y devuelve respuestas.
Frontend Backend
Lo que el usuario ve Lo que el usuario NO ve
Corre en el navegador del usuario Corre en un servidor (otra computadora)
HTML, CSS, JavaScript Node.js, Python, PHP, Java, etc.
Diseño, botones, formularios Lógica de negocio, base de datos, seguridad
En Aroma: index.html, styles.css, app.js En Aroma: servidor Express + Supabase

¿Qué es una Base de Datos?

Analogía de la libreta:
En un restaurante sin computadoras, el dueño tiene una libreta donde anota: qué productos tiene, a qué precio, cuántos pedidos se hicieron hoy, quiénes son sus empleados.

Una base de datos es esa libreta, pero digital, organizada en tablas (como hojas de Excel), que puede almacenar millones de registros y consultarlos en milisegundos.

En Aroma tendremos tablas como:

  • productos — id, nombre, precio, categoría, imagen
  • pedidos — id, cliente, total, estado, fecha
  • usuarios — id, email, contraseña cifrada, rol

El flujo completo: ¿qué pasa cuando haces clic en "Ver Menú"?

1
El usuario hace clic en el botón "Ver Menú" en su navegador.
2
JavaScript detecta el clic y manda una petición HTTP al servidor: "Dame la lista de productos".
3
El servidor (Node.js/Express) recibe la petición y le pregunta a la base de datos: "Dame todos los productos activos".
4
La base de datos (Supabase) busca los productos y los devuelve al servidor en formato JSON.
5
El servidor responde al navegador con los datos de los productos.
6
JavaScript en el navegador recibe los datos y dibuja las tarjetas de productos en la pantalla.
7
El usuario ve el menú con todos los productos. ¡Todo en menos de un segundo!
Dato clave: Todo esto lo vas a implementar tú mismo en este curso. Al final, entenderás exactamente qué ocurre en cada uno de estos 7 pasos.

2 Node.js

¿Qué es? Node.js es un programa que permite ejecutar JavaScript fuera del navegador, directamente en tu computadora o en un servidor. Antes de Node.js (creado en 2009), JavaScript solo podía correr en navegadores.

¿Por qué lo necesitamos? Porque nuestro backend (el servidor de Cafetería Aroma) está escrito en JavaScript usando Node.js. También necesitamos Node para usar npm, el gestor de paquetes que nos permite instalar librerías como Express.

1
Ve a https://nodejs.org. Verás dos versiones: LTS y "Current". Descarga siempre la versión LTS (Long Term Support = Soporte a Largo Plazo). Es la más estable y la recomendada para proyectos serios.
2
Instala con todas las opciones por defecto. El instalador también instalará npm automáticamente.
3
Verifica la instalación. Abre una terminal (en Windows: busca "CMD" o "PowerShell" en el menú de inicio) y ejecuta: 
node --version
npm --version
Deberías ver algo como v20.11.0 y 10.2.4. Los números exactos pueden variar, lo importante es que no dé error.
¿Qué es npm? npm (Node Package Manager) es el gestor de paquetes de Node.js. Es como una "tienda de herramientas" gratuita: con un solo comando puedes instalar cualquiera de los más de 2 millones de librerías disponibles. Por ejemplo: npm install express instala el framework Express.

3 Git

¿Qué es? Git es un sistema de control de versiones. Registra todos los cambios que haces en tu código a lo largo del tiempo, permitiéndote volver a cualquier versión anterior, trabajar en equipo sin pisarte con otros, y tener un historial completo de tu proyecto.

1
Ve a https://git-scm.com y descarga Git para tu sistema operativo.
2
Durante la instalación, cuando te pregunte por el editor por defecto, selecciona Visual Studio Code. Para el resto, las opciones por defecto están bien.
3
Configura tu identidad (Git necesita saber quién hace los cambios). Abre la terminal y ejecuta estos dos comandos, reemplazando con tus datos: 
git config --global user.name "Tu Nombre Completo"
git config --global user.email "tu@email.com"
4
Verifica la instalación: 
git --version
Deberías ver algo como git version 2.43.0.

4 Google Chrome

¿Por qué Chrome? Chrome es el navegador más usado en el mundo y tiene las mejores DevTools (herramientas de desarrollo), que son fundamentales para debuggear (encontrar y corregir errores) en páginas web.

1
Descarga Chrome desde https://www.google.com/chrome si no lo tienes instalado.
2
Aprende a abrir las DevTools: presiona F12 (o Ctrl+Shift+I) en cualquier página web. Verás un panel con varias pestañas:
  • Elements: Muestra el HTML de la página. Puedes hacer clic en cualquier elemento para inspeccionarlo.
  • Console: Muestra errores de JavaScript y te permite ejecutar código directamente.
  • Network: Muestra todas las peticiones HTTP que hace la página (muy útil para ver si el API funciona).
  • Application: Muestra cookies, localStorage y otros datos guardados en el navegador.
Truco de desarrollador: Con Chrome DevTools abierto, haz clic derecho en cualquier elemento de cualquier página web y elige "Inspeccionar". Podrás ver cómo está construida esa página. ¡Es una de las mejores formas de aprender!

7. Primer contacto con Git

Git es posiblemente la herramienta más importante de un desarrollador, después de su editor de código. Vamos a entenderlo bien desde el principio.

Analogía del "control de versiones":
¿Alguna vez has escrito un documento Word y guardado copias como "informe_final.docx", "informe_final_v2.docx", "informe_final_ESTE_SI.docx"? Eso es un control de versiones manual, ¡y es un caos!

Git hace eso de forma inteligente y automática. Cada vez que terminas una parte de tu código y lo "guardas" en Git (con un commit), Git recuerda exactamente cómo estaban todos los archivos en ese momento. Puedes tener miles de "versiones" y Git las gestiona sin que tengas miles de archivos duplicados.

Es como un Ctrl+Z infinito para todo el proyecto, con descripciones de qué hiciste en cada versión.

Conceptos clave de Git

  • Repositorio (repo): Una carpeta de proyecto controlada por Git. Contiene todos tus archivos y el historial completo de cambios.
  • Commit: Un "punto de guardado" con nombre. Es una foto instantánea de todos tus archivos en un momento dado.
  • Branch (rama): Una línea de desarrollo paralela. Puedes crear una rama para probar algo nuevo sin afectar el código principal.
  • Remote: Una copia del repositorio en internet (como GitHub). Es tu respaldo en la nube.

Los comandos Git que más usarás

Configurar un nuevo repositorio

# Inicializar Git en una carpeta (solo se hace una vez al inicio del proyecto)
git init

# Vincular con un repositorio de GitHub
git remote add origin https://github.com/tu-usuario/cafeteria-aroma.git

El ciclo diario de trabajo con Git

# 1. Ver qué archivos cambiaron
git status

# 2. Agregar los archivos que quieres guardar en el commit
git add .          # Agrega TODOS los archivos modificados
git add index.html # Agrega solo ese archivo específico

# 3. Crear el commit (el "punto de guardado") con un mensaje descriptivo
git commit -m "Agrego la página del menú con filtros por categoría"

# 4. Subir los cambios a GitHub
git push origin main

Comandos de consulta

# Ver el historial de commits (quién hizo qué y cuándo)
git log

# Ver los cambios que hiciste desde el último commit
git diff

# Descargar los últimos cambios de GitHub (útil si trabajas en equipo)
git pull origin main
Regla de oro de Git: Haz commits frecuentes y con mensajes descriptivos. Un buen mensaje de commit empieza con un verbo en presente: "Agrego...", "Corrijo...", "Actualizo...", "Elimino...". Así, cuando veas el historial, sabes exactamente qué cambió en cada paso.
¿Metiste la pata? Si rompiste algo y quieres volver al último commit guardado, puedes usar git checkout -- . (con cuidado, esto descarta todos los cambios sin guardar). Si el commit ya fue hecho, git revert crea un nuevo commit que deshace el anterior sin borrar el historial.

├── database/                  ← Todo relacionado con la base de datos
│   ├── schema.sql             ← Estructura de las tablas (CREATE TABLE...)
│   └── seed.sql               ← Datos iniciales (productos de ejemplo)
│
└── docs/                      ← Documentación del proyecto
    ├── guias-didacticas/      ← ¡Estás aquí! Las guías de aprendizaje
    └── guias/                 ← Guías técnicas de referencia

¿Para qué sirve cada parte?

📁
frontend/ — Todo lo que va al navegador del cliente. Los archivos de esta carpeta son descargados por el navegador del usuario cuando visita la página. NO deben contener contraseñas ni información confidencial.
📁
backend/ — El servidor Node.js que corre en un servidor remoto (en Render). Maneja la lógica de negocio, verifica permisos, y es el único que puede hablar directamente con la base de datos.
📁
database/ — Los archivos SQL que definen la estructura de la base de datos. Son como el "plano" de cómo están organizadas las tablas, relaciones y datos iniciales.
📁
docs/ — Documentación, guías y notas sobre el proyecto. Muy importante en proyectos reales para que cualquier persona pueda entender el sistema.
Principio de separación de responsabilidades: Cada carpeta tiene una responsabilidad clara. Frontend = presentación, Backend = lógica, Database = datos. Esta separación hace que si necesitamos cambiar la base de datos, no tengamos que tocar el frontend, y viceversa.

Ejercicio Final: Verifica tu Entorno de Desarrollo

Antes de avanzar a la siguiente guía, verifica que todo está instalado correctamente siguiendo estos pasos:

1
Abre una terminal (CMD, PowerShell o la terminal de VS Code con Ctrl+`) y ejecuta los siguientes comandos. Cada uno debe mostrar un número de versión sin errores: 
node --version
npm --version
git --version
2
Crea una carpeta de práctica llamada mi-primer-proyecto en el escritorio: 
# En la terminal:
mkdir Desktop/mi-primer-proyecto
cd Desktop/mi-primer-proyecto
3
Abre esa carpeta en VS Code: 
code .
(El punto . significa "la carpeta actual"). VS Code debería abrirse con esa carpeta como proyecto.
4
Inicializa un repositorio Git en esa carpeta. Desde la terminal de VS Code: 
git init
Deberías ver: Initialized empty Git repository in .../mi-primer-proyecto/.git/
5
Crea un archivo de texto simple. En VS Code, usa Ctrl+N para crear un archivo nuevo, escribe "Hola, soy mi primer proyecto web", y guárdalo como README.md con Ctrl+S.
6
Haz tu primer commit: 
git add .
git commit -m "Inicio mi primer proyecto web"
Deberías ver un mensaje confirmando que el commit fue creado.
