1. ¿Qué es JWT y cómo funciona la autenticación?

Analogía del pulsera de evento: Cuando entras a un concierto, te dan una pulsera (el JWT token) que prueba que pagaste. En cada área VIP o barra, el guardia revisa tu pulsera sin tener que llamar a la taquilla otra vez. El token JWT funciona igual: Supabase lo entrega al hacer login, y el backend lo verifica en cada petición sin consultar la base de datos de autenticación cada vez.

JWT (JSON Web Token) es un estándar para transmitir información entre partes de forma segura. Un token JWT tiene tres partes separadas por puntos:

  • Header: El algoritmo de firma usado (ej: HS256)
  • Payload: Los datos: el ID del usuario, cuándo expira, etc.
  • Signature: Una firma criptográfica que garantiza que nadie modificó el token
// Un token JWT se ve así:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyLWlkLTEyMyIsImVtYWlsIjoiY2xpZW50ZUBhcm9tYS5jb20iLCJleHAiOjE3MzAwMDAwMDB9.xK9d3mP...

// Decodificado (en jwt.io puedes verlo):
// Header: { "alg": "HS256", "typ": "JWT" }
// Payload: { "sub": "user-id-123", "email": "cliente@aroma.com", "exp": 1730000000 }
// Signature: verificada por Supabase

El flujo de autenticación en Aroma

1
El usuario hace login en el frontend → Supabase verifica email/contraseña y devuelve un token JWT.
2
El frontend guarda el token (en memoria o sessionStorage).
3
En cada petición al backend, el frontend incluye el token en el header: Authorization: Bearer <token>.
4
El middleware requireAuth extrae el token, lo verifica con Supabase, y adjunta el usuario a req.user.
5
El controlador usa req.user.id y req.user.role para hacer su trabajo.

2. El middleware de autenticación

Este es el archivo más importante de seguridad en el backend. Cada vez que alguien intenta acceder a una ruta protegida, este middleware se ejecuta primero.

// backend/src/middleware/auth.js
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
    process.env.SUPABASE_URL,
    process.env.SUPABASE_SERVICE_ROLE_KEY
);

// Middleware: verificar que el usuario está autenticado
export async function requireAuth(req, res, next) {
    try {
        // Paso 1: Leer el header Authorization
        const authHeader = req.headers.authorization;

        if (!authHeader || !authHeader.startsWith('Bearer ')) {
            return res.status(401).json({
                error: 'Token de autenticación requerido'
            });
        }

        // Paso 2: Extraer el token
        // "Bearer eyJ..." → "eyJ..."
        const token = authHeader.split(' ')[1];

        // Paso 3: Verificar el token con Supabase Auth
        const { data: { user }, error } = await supabase.auth.getUser(token);

        if (error || !user) {
            return res.status(401).json({ error: 'Token inválido o expirado' });
        }

        // Paso 4: Obtener el perfil del usuario (contiene el rol)
        const { data: profile } = await supabase
            .from('profiles')
            .select('role, active')
            .eq('id', user.id)
            .single();

        // Verificar que la cuenta está activa
        if (!profile?.active) {
            return res.status(403).json({ error: 'Cuenta desactivada' });
        }

        // Paso 5: Adjuntar el usuario al request para los controladores
        req.user = { ...user, role: profile?.role || 'client' };

        next(); // Todo bien, continuar al controlador

    } catch (error) {
        console.error('Error en middleware de autenticación:', error);
        res.status(500).json({ error: 'Error de autenticación' });
    }
}

// Middleware: verificar que el usuario tiene el rol adecuado
// Uso: requireRole('cashier', 'admin') acepta cajeros Y admins
export function requireRole(...roles) {
    return (req, res, next) => {
        // Esta función debe usarse DESPUÉS de requireAuth
        if (!req.user) {
            return res.status(401).json({ error: 'No autenticado' });
        }

        if (!roles.includes(req.user.role)) {
            return res.status(403).json({
                error: `Acceso denegado. Rol requerido: ${roles.join(' o ')}`
            });
        }

        next();
    };
}

La diferencia entre 401 y 403

Código Nombre Significado Ejemplo en Aroma
401 Unauthorized "No sé quién eres" — falta o es inválido el token Intentar ver pedidos sin haber iniciado sesión
403 Forbidden "Sé quién eres, pero no tienes permiso" Un cliente intentando acceder al panel del cajero
Tip: Devolver 401 cuando no hay token es correcto porque le dice al cliente "por favor, autentica tu petición". Devolver 403 cuando el usuario existe pero no tiene el rol adecuado le dice "ya sé quién eres, pero esto no te corresponde". Usar el código correcto facilita el debugging.

3. Rutas protegidas con middleware

Ahora usamos los middlewares en los archivos de rutas para proteger los endpoints según el nivel de acceso requerido:

// backend/src/routes/client.js
// Rutas para clientes autenticados
import { Router } from 'express';
import { requireAuth } from '../middleware/auth.js';
import { createOrder, getMyOrders } from '../controllers/ordersController.js';

const router = Router();

// Aplicar autenticación a TODAS las rutas de este router
router.use(requireAuth);

router.post('/orders', createOrder);       // Crear un pedido
router.get('/my-orders', getMyOrders);     // Ver mis pedidos

export default router;
// backend/src/routes/cashier.js
// Rutas para el panel del cajero
import { Router } from 'express';
import { requireAuth, requireRole } from '../middleware/auth.js';
import { getCashierOrders, updateOrderStatus } from '../controllers/ordersController.js';

const router = Router();

// Primero verificar que está autenticado,
// luego verificar que tiene rol de cajero o admin
router.use(requireAuth);
router.use(requireRole('cashier', 'admin'));

router.get('/orders', getCashierOrders);
router.patch('/orders/:id/status', updateOrderStatus);

export default router;
// backend/src/routes/admin.js
// Rutas para el panel del administrador
import { Router } from 'express';
import { requireAuth, requireRole } from '../middleware/auth.js';
import { getDashboard, createProduct, updateProduct, deleteProduct } from '../controllers/adminController.js';

const router = Router();

// Solo administradores
router.use(requireAuth);
router.use(requireRole('admin'));

router.get('/dashboard', getDashboard);
router.post('/products', createProduct);
router.put('/products/:id', updateProduct);
router.patch('/products/:id', updateProduct);
router.delete('/products/:id', deleteProduct);

export default router;
Analogía de la cafetería: En una cafetería real, hay puertas con acceso restringido. La puerta de la cocina solo la puede abrir el personal (cajeros y cocineros). La oficina del gerente, solo el gerente. Y la sala de clientes es de acceso libre. Los middlewares requireAuth y requireRole son esas puertas con llaves.

4. El controlador de pedidos — createOrder

Este es uno de los controladores más complejos porque implementa una regla de seguridad crítica: nunca confiar en los precios que envía el cliente.

// backend/src/controllers/ordersController.js
import supabase from '../services/supabase.js';

// POST /api/orders
export async function createOrder(req, res) {
    try {
        const userId = req.user.id; // Viene del middleware requireAuth

        // Extraer datos del body
        const {
            customer_name,
            customer_phone,
            delivery_address,
            payment_method,
            notes,
            items  // Array: [{ product_id: "uuid", quantity: 2 }, ...]
        } = req.body;

        // ===== VALIDACIONES =====

        if (!customer_name || !customer_phone || !delivery_address) {
            return res.status(400).json({
                error: 'Nombre, teléfono y dirección son requeridos'
            });
        }

        if (!items || items.length === 0) {
            return res.status(400).json({
                error: 'El pedido debe tener al menos un item'
            });
        }

        // Validar que las cantidades sean positivas
        for (const item of items) {
            if (!item.quantity || item.quantity <= 0) {
                return res.status(400).json({
                    error: 'Todas las cantidades deben ser mayores a 0'
                });
            }
        }

        // ===== VERIFICAR PRODUCTOS EN LA BASE DE DATOS =====
        // Obtenemos los precios REALES desde la BD, NO del cliente
        const productIds = items.map(item => item.product_id);
        const { data: products, error: productsError } = await supabase
            .from('products')
            .select('id, nombre, precio, activo')
            .in('id', productIds);

        if (productsError) throw productsError;

        // Verificar que todos los productos existen y están activos
        for (const item of items) {
            const product = products.find(p => p.id === item.product_id);
            if (!product) {
                return res.status(400).json({
                    error: `Producto no encontrado: ${item.product_id}`
                });
            }
            if (!product.activo) {
                return res.status(400).json({
                    error: `Producto no disponible: ${product.nombre}`
                });
            }
        }

        // ===== CALCULAR TOTAL CON PRECIOS REALES =====
        const orderItems = items.map(item => {
            const product = products.find(p => p.id === item.product_id);
            return {
                product_id: product.id,
                product_name: product.nombre,
                unit_price: product.precio,        // Precio de la BD, no del cliente
                quantity: item.quantity,
                subtotal: product.precio * item.quantity
            };
        });

        const total = orderItems.reduce((sum, item) => sum + item.subtotal, 0);

        // ===== CREAR EL PEDIDO =====
        const { data: order, error: orderError } = await supabase
            .from('orders')
            .insert({
                user_id: userId,
                customer_name,
                customer_phone,
                delivery_address,
                payment_method,
                notes,
                total,
                status: 'pending'
                // El trigger de la BD genera el código (ej: ARO-0042)
            })
            .select()
            .single();

        if (orderError) throw orderError;

        // ===== INSERTAR LOS ITEMS DEL PEDIDO =====
        const { error: itemsError } = await supabase
            .from('order_items')
            .insert(
                orderItems.map(item => ({ ...item, order_id: order.id }))
            );

        if (itemsError) throw itemsError;

        // Responder con el pedido creado
        res.status(201).json({
            order,
            message: `Pedido ${order.code} creado exitosamente`
        });

    } catch (error) {
        console.error('Error creando pedido:', error);
        res.status(500).json({ error: 'Error al crear el pedido' });
    }
}
Principio de seguridad critico — Nunca confiar en los precios del cliente: Un usuario malintencionado podría modificar el JavaScript del frontend para enviar { product_id: "latte-id", quantity: 1, price: 0.01 }. Si el servidor aceptara ese precio, el usuario compraría un café por un centavo. El backend siempre obtiene los precios directamente de la base de datos, ignorando cualquier precio que venga en la petición.

GET /api/my-orders — Ver mis propios pedidos

// GET /api/my-orders
export async function getMyOrders(req, res) {
    try {
        const userId = req.user.id;

        const { data: orders, error } = await supabase
            .from('orders')
            .select(`
                id, code, status, total, created_at,
                customer_name, payment_method, notes,
                items:order_items(product_name, unit_price, quantity, subtotal)
            `)
            .eq('user_id', userId)           // Solo los pedidos de ESTE usuario
            .order('created_at', { ascending: false })
            .limit(20);

        if (error) throw error;

        res.json({ orders });

    } catch (error) {
        res.status(500).json({ error: 'Error al obtener tus pedidos' });
    }
}
Tip — Filtrar por user_id: Aunque podría parecer que la RLS de Supabase ya protege esto, cuando usamos la SERVICE_ROLE_KEY en el backend, la RLS está desactivada. Por eso es fundamental filtrar explícitamente con .eq('user_id', userId) donde userId viene del token verificado (no del cliente). Así garantizamos que cada usuario solo ve sus propios pedidos.

5. Panel del Cajero — Backend

El cajero necesita ver los pedidos activos, filtrarlos por estado o fecha, y poder cambiar su estado conforme avanza la preparación.

// GET /api/cashier/orders
export async function getCashierOrders(req, res) {
    try {
        const { status, fecha } = req.query;

        let query = supabase
            .from('orders')
            .select(`
                id, code, status, customer_name, customer_phone,
                delivery_address, payment_method, total, notes,
                created_at,
                items:order_items(
                    id, product_name, unit_price, quantity, subtotal
                )
            `)
            .order('created_at', { ascending: false });

        // Filtrar por estado si se especificó
        if (status) {
            // Soporte para múltiples estados: ?status=pending,confirmed
            const estados = status.split(',');
            query = query.in('status', estados);
        }

        // Filtrar por fecha si se especificó: ?fecha=2024-11-15
        if (fecha) {
            const inicio = `${fecha}T00:00:00`;
            const fin = `${fecha}T23:59:59`;
            query = query.gte('created_at', inicio).lte('created_at', fin);
        }

        const { data: orders, error } = await query;
        if (error) throw error;

        res.json({ orders });

    } catch (error) {
        res.status(500).json({ error: 'Error al obtener pedidos' });
    }
}

// PATCH /api/cashier/orders/:id/status
export async function updateOrderStatus(req, res) {
    try {
        const { id } = req.params;
        const { status } = req.body;

        // Validar que el estado es uno de los permitidos
        const validStatuses = ['pending', 'confirmed', 'preparing', 'ready', 'delivered', 'cancelled'];
        if (!validStatuses.includes(status)) {
            return res.status(400).json({
                error: `Estado inválido. Valores permitidos: ${validStatuses.join(', ')}`
            });
        }

        const { data: order, error } = await supabase
            .from('orders')
            .update({
                status,
                updated_at: new Date().toISOString()
            })
            .eq('id', id)
            .select()
            .single();

        if (error) throw error;

        if (!order) {
            return res.status(404).json({ error: 'Pedido no encontrado' });
        }

        res.json({
            order,
            message: `Estado del pedido ${order.code} actualizado a: ${status}`
        });

    } catch (error) {
        res.status(500).json({ error: 'Error al actualizar el pedido' });
    }
}

Los estados de un pedido en Aroma

Estado Descripción Quién lo cambia
pending Recién creado, esperando confirmación Sistema (automático al crear)
confirmed Cajero confirmó que es válido Cajero
preparing En preparación en la cocina Cajero
ready Listo para entregar Cajero
delivered Entregado al cliente Cajero
cancelled Cancelado Cajero o Admin

6. CRUD de Productos para el Administrador

El administrador necesita gestionar el catálogo completo: crear, editar y desactivar productos.

// backend/src/controllers/adminController.js
import supabase from '../services/supabase.js';

// POST /api/admin/products — Crear producto
export async function createProduct(req, res) {
    try {
        const { nombre, descripcion, precio, costo, category_id, imagen_url } = req.body;

        // Validar campos obligatorios
        if (!nombre || !precio || !category_id) {
            return res.status(400).json({
                error: 'Nombre, precio y categoría son requeridos'
            });
        }

        // Validar que el precio es un número positivo
        if (isNaN(precio) || precio <= 0) {
            return res.status(400).json({ error: 'El precio debe ser un número positivo' });
        }

        const { data: product, error } = await supabase
            .from('products')
            .insert({
                nombre,
                descripcion,
                precio: parseFloat(precio),
                costo: costo ? parseFloat(costo) : null,
                category_id,
                imagen_url,
                activo: true
            })
            .select()
            .single();

        if (error) throw error;

        res.status(201).json({ product });

    } catch (error) {
        console.error('Error creando producto:', error);
        res.status(500).json({ error: 'Error al crear producto' });
    }
}

// PUT /api/admin/products/:id — Actualizar producto
export async function updateProduct(req, res) {
    try {
        const { id } = req.params;
        const updates = req.body;

        // Asegurar que no se pueda modificar el ID
        delete updates.id;

        const { data: product, error } = await supabase
            .from('products')
            .update({
                ...updates,
                updated_at: new Date().toISOString()
            })
            .eq('id', id)
            .select()
            .single();

        if (error) throw error;

        if (!product) {
            return res.status(404).json({ error: 'Producto no encontrado' });
        }

        res.json({ product });

    } catch (error) {
        res.status(500).json({ error: 'Error al actualizar producto' });
    }
}

// DELETE /api/admin/products/:id — Soft delete (desactivar)
export async function deleteProduct(req, res) {
    try {
        const { id } = req.params;

        // En lugar de borrar físicamente, desactivamos el producto
        // Esto preserva el historial de pedidos que referencian este producto
        const { data: product, error } = await supabase
            .from('products')
            .update({ activo: false, updated_at: new Date().toISOString() })
            .eq('id', id)
            .select()
            .single();

        if (error) throw error;

        res.json({ message: `Producto "${product.nombre}" desactivado` });

    } catch (error) {
        res.status(500).json({ error: 'Error al eliminar producto' });
    }
}
¿Por qué "soft delete" y no borrar directamente? Si borramos un producto de la base de datos, los pedidos históricos que lo contengan quedarían con referencias rotas (foreign key violation). Al solo desactivarlo (activo: false), preservamos la integridad del historial. El producto simplemente deja de mostrarse en el menú, pero los registros pasados siguen siendo válidos.

7. El cliente de Supabase en el backend

Para centralizar la instancia de Supabase y no crearla en cada archivo, usamos un módulo de servicio:

// backend/src/services/supabase.js
import { createClient } from '@supabase/supabase-js';

// Verificar que las variables de entorno están configuradas
if (!process.env.SUPABASE_URL || !process.env.SUPABASE_SERVICE_ROLE_KEY) {
    throw new Error(
        'Faltan variables de entorno de Supabase. ' +
        'Verifica que el archivo .env existe y tiene SUPABASE_URL y SUPABASE_SERVICE_ROLE_KEY'
    );
}

// Crear cliente con la SERVICE_ROLE_KEY (acceso total, solo para backend)
const supabase = createClient(
    process.env.SUPABASE_URL,
    process.env.SUPABASE_SERVICE_ROLE_KEY,
    {
        auth: {
            autoRefreshToken: false,
            persistSession: false
        }
    }
);

export default supabase;
Diferencia SERVICE_ROLE_KEY vs ANON_KEY:
  • ANON_KEY (frontend): Respeta las políticas RLS. Un usuario con esta clave solo puede ver lo que las reglas le permiten.
  • SERVICE_ROLE_KEY (backend): Salta todas las políticas RLS. Tiene acceso total como administrador de la BD. Por eso NUNCA debe exponerse al cliente.

8. Probando las rutas protegidas con Thunder Client

Para probar los endpoints que requieren autenticación, necesitas obtener primero un token JWT válido.

Paso 1: Obtener el token de Supabase

// Hacer POST directamente a la API de autenticación de Supabase
// URL: https://TU-PROYECTO.supabase.co/auth/v1/token?grant_type=password
// Body (JSON):
{
    "email": "cajero@aroma.com",
    "password": "tu-contraseña"
}

// La respuesta incluye:
{
    "access_token": "eyJhbGci...",   // ← Este es el JWT
    "token_type": "bearer",
    "expires_in": 3600,
    "refresh_token": "..."
}

Paso 2: Usar el token en Thunder Client

1
Crear una nueva petición en Thunder Client.
2
Ir a la pestaña "Auth" → seleccionar "Bearer".
3
Pegar el access_token en el campo "Token".
4
Hacer la petición. Thunder Client agregará automáticamente el header Authorization: Bearer eyJ...
Ojo con la expiración: Los tokens de Supabase expiran en 1 hora por defecto. Si recibes un 401 después de trabajar un rato, simplemente repite el login para obtener un token fresco.

Resumen de la Guía 12

  • JWT es un token que el cliente presenta en cada petición para demostrar su identidad.
  • El token se envía en el header Authorization: Bearer <token>.
  • El middleware requireAuth verifica el token y adjunta el usuario a req.user.
  • requireRole() protege rutas por rol: client, cashier, admin.
  • Error 401 = no autenticado (sin token). Error 403 = sin permiso (token válido pero rol incorrecto).
  • El controlador createOrder nunca confía en los precios del frontend; siempre los obtiene de la BD.
  • El soft delete desactiva productos en lugar de borrarlos para preservar el historial.
  • La SERVICE_ROLE_KEY salta la RLS — solo debe usarse en el backend.
