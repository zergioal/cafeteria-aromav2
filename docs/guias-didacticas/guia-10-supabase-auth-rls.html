1. ¿Qué es la autenticación?

Antes de hablar de código, necesitamos entender dos conceptos que se confunden frecuentemente:

Concepto Pregunta que responde Ejemplo en Aroma
Autenticación ¿Quién eres tú? El usuario ingresa email y contraseña; verificamos que es quien dice ser.
Autorización ¿Qué puedes hacer? Un cajero puede ver pedidos pero no puede editar productos del menú.
Analogía: El edificio de oficinas
Imagina que llegas a trabajar a un edificio corporativo. En la entrada, la recepcionista te pide tu DNI y verifica que estás en la lista de empleados — eso es la autenticación. Una vez dentro, tu tarjeta de acceso determina a qué pisos y qué salas puedes entrar — eso es la autorización. Puedes estar perfectamente identificado pero sin permiso de entrar a la sala del servidor.

En Aroma necesitamos ambas capas:

  • Autenticación: solo usuarios registrados pueden hacer pedidos.
  • Autorización: solo admins pueden modificar el menú, solo cajeros pueden cambiar el estado de los pedidos.

2. Supabase Auth — Autenticación lista para usar

Implementar autenticación desde cero es complejo y riesgoso: hay que manejar el hash de contraseñas, tokens de sesión, expiración, renovación, recuperación de contraseña, etc. Supabase Auth hace todo eso por nosotros.

¿Qué hace Supabase Auth por nosotros?

  • Nunca guarda contraseñas en texto plano. Las procesa con bcrypt (algoritmo de hash unidireccional). Ni siquiera el equipo de Supabase puede ver tu contraseña.
  • Genera JWT tokens para cada sesión iniciada.
  • Maneja el refresh automático de tokens antes de que expiren.
  • Permite login con email/contraseña o con proveedores OAuth (Google, GitHub, Apple, etc.).
  • Envía emails de confirmación y recuperación de contraseña.

El módulo auth.js de Aroma

Veamos cómo el frontend de Aroma usa Supabase Auth:

// frontend/js/auth.js
import { createClient } from '@supabase/supabase-js';

// Crear el cliente de Supabase (solo para auth en el frontend)
// NOTA: Usamos la ANON KEY (clave pública), nunca la service_role key aquí
const supabase = createClient(
    CONFIG.SUPABASE_URL,
    CONFIG.SUPABASE_ANON_KEY
);

const Auth = {
    // Estado del usuario actual en memoria
    currentUser: null,
    currentSession: null,

    // ============================================================
    // Registrar un nuevo usuario
    async signUp(email, password, fullName) {
        const { data, error } = await supabase.auth.signUp({
            email,
            password,
            options: {
                data: { full_name: fullName }
                // Estos datos extra se guardan en user_metadata
                // El trigger handle_new_user los leerá para crear el perfil
            }
        });

        if (error) throw error;
        return data;
        // Supabase envía un email de confirmación automáticamente
    },

    // ============================================================
    // Iniciar sesión
    async signIn(email, password) {
        const { data, error } = await supabase.auth.signInWithPassword({
            email,
            password
        });

        if (error) throw error;

        // Guardar la sesión en el estado y en localStorage
        this.currentUser    = data.user;
        this.currentSession = data.session;
        localStorage.setItem('aroma_session', JSON.stringify(data.session));

        return data;
        // data.session contiene el JWT (access_token) y el refresh_token
    },

    // ============================================================
    // Cerrar sesión
    async signOut() {
        await supabase.auth.signOut();
        // Limpiar el estado local
        this.currentUser    = null;
        this.currentSession = null;
        localStorage.removeItem('aroma_session');
        Cart.clear(); // Vaciar el carrito al cerrar sesión
    },

    // ============================================================
    // Obtener la sesión actual (al cargar la página)
    async getSession() {
        const { data: { session } } = await supabase.auth.getSession();
        this.currentSession = session;
        this.currentUser    = session?.user || null;
        return session;
        // Si el token expiró, Supabase lo renueva automáticamente usando
        // el refresh_token guardado en el navegador
    },

    // ============================================================
    // ¿El usuario está logueado?
    isLoggedIn() {
        return !!this.currentUser;
    },

    // ============================================================
    // ¿Qué rol tiene el usuario?
    getRole() {
        // Primero busca en user_metadata (lo que el usuario puede modificar)
        // Luego en app_metadata (lo que solo el admin puede modificar — más confiable)
        return this.currentUser?.app_metadata?.role
            || this.currentUser?.user_metadata?.role
            || 'client'; // Por defecto, todos son clientes
    },

    // ============================================================
    // Escuchar cambios de autenticación en tiempo real
    // Útil para actualizar la UI cuando el usuario hace login/logout
    onAuthStateChange(callback) {
        supabase.auth.onAuthStateChange((event, session) => {
            this.currentSession = session;
            this.currentUser    = session?.user || null;
            callback(event, session);
            // event puede ser: SIGNED_IN, SIGNED_OUT, TOKEN_REFRESHED, etc.
        });
    }
};
Tip: El operador ?. (optional chaining)
En el código ves session?.user. El operador ?. significa "si session existe, dame session.user; si session es null o undefined, devuelve undefined en lugar de lanzar un error". Es muy útil cuando trabajas con datos que pueden no existir.

3. JWT Tokens — La llave de acceso digital

Analogía: El ticket de cine infalsificable
Un JWT es como un ticket de cine sofisticado: contiene información impresa (a qué función vas, en qué asiento, cuándo vence) y tiene un sello holográfico que nadie puede falsificar sin la máquina del cine. Puedes leer el ticket y saber toda esa información, pero no puedes fabricar uno falso.

JWT significa JSON Web Token. Es un estándar para transmitir información de forma segura entre partes. Un JWT tiene tres partes separadas por puntos:

eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjoiMTIzIn0.SflKxwRJSMeKKF2QT4fw

    ↑ HEADER         ↑ PAYLOAD                  ↑ FIRMA DIGITAL

HEADER (decodificado):
{
    "alg": "HS256",   ← Algoritmo de firma usado
    "typ": "JWT"
}

PAYLOAD (decodificado):
{
    "sub":   "uuid-del-usuario",    ← ID del usuario (subject)
    "email": "juan@ejemplo.com",
    "role":  "client",
    "iat":   1710000000,            ← Issued At: cuándo se creó
    "exp":   1710003600             ← Expiration: cuándo vence (1 hora después)
}

FIRMA DIGITAL:
    HMAC-SHA256(base64(header) + "." + base64(payload), CLAVE_SECRETA)
    ← Solo Supabase conoce la clave secreta para generar y verificar esto

Flujo completo del JWT en Aroma

1. Usuario hace login → frontend envía email + password a Supabase Auth
2. Supabase verifica credenciales y genera un JWT (access_token)
3. Frontend recibe el JWT y lo guarda en localStorage
4. En cada petición a la API, frontend incluye el token:
    Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
5. El backend (o Supabase RLS) verifica el JWT con su clave secreta
6. Si es válido, extrae el userID y el rol del payload
7. Autoriza o rechaza la operación según los permisos
Importante: El JWT no es secreto, pero es infalsificable
Cualquiera puede leer el contenido de un JWT (está codificado en base64, no cifrado). Sin embargo, nadie puede modificarlo sin invalidar la firma. Si un atacante intenta cambiar su role de "client" a "admin" en el payload, la firma digital ya no coincide y el servidor lo rechaza inmediatamente.

4. Row Level Security (RLS) — Seguridad a nivel de filas

Analogía: El archivero inteligente
Imagina una oficina donde hay un enorme archivero con las carpetas de todos los clientes. Sin seguridad, cualquier empleado que llegue puede abrir cualquier cajón y ver los datos de cualquier cliente. Con RLS, hay un sistema automático que, antes de abrir cualquier cajón, verifica: "¿Este empleado tiene permiso para ver estos archivos?" Si no lo tiene, el cajón no se abre, aunque la persona ya esté dentro del edificio.

RLS es una característica de PostgreSQL que permite definir políticas de acceso a nivel de cada fila. En lugar de restringir solo qué tablas puede ver un usuario, podemos definir exactamente qué filas dentro de esas tablas puede leer o modificar.

¿Por qué es tan importante?

Sin RLS, si un cliente logra acceder a la API de Supabase, podría ejecutar:

SELECT * FROM orders;  -- ¡Vería los pedidos de TODOS los clientes!

Con RLS activado y la política correcta, esa misma consulta solo devuelve los pedidos del usuario autenticado. La restricción ocurre dentro de la base de datos, no en el código.

Activar RLS en las tablas

Por defecto, RLS está desactivado. Al activarlo, nadie puede acceder a nada hasta que creemos políticas explícitas:

-- Activar RLS en cada tabla (bloquea todo el acceso por defecto)
ALTER TABLE products      ENABLE ROW LEVEL SECURITY;
ALTER TABLE categories    ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders        ENABLE ROW LEVEL SECURITY;
ALTER TABLE order_items   ENABLE ROW LEVEL SECURITY;
ALTER TABLE profiles      ENABLE ROW LEVEL SECURITY;

Crear Políticas (Policies)

Una política define quién puede hacer qué en una tabla. Tiene tres componentes clave:

  • FOR SELECT / INSERT / UPDATE / DELETE / ALL: qué operación controla.
  • USING (...): condición para SELECT y DELETE (qué filas son visibles o pueden eliminarse).
  • WITH CHECK (...): condición para INSERT y UPDATE (qué filas pueden crearse o modificarse).
-- Política: Cualquiera puede ver los productos activos (menú público)
-- No necesita autenticación: el restaurante es público
CREATE POLICY "ver_productos_activos"
ON products
FOR SELECT
USING (activo = true);
-- USING = la condición que debe cumplir la FILA para ser devuelta

-- Política: Solo admins pueden modificar productos
CREATE POLICY "admin_modifica_productos"
ON products
FOR ALL  -- Controla INSERT, UPDATE y DELETE
USING (
    EXISTS (
        SELECT 1 FROM profiles
        WHERE profiles.id = auth.uid()  -- auth.uid() = ID del usuario autenticado
        AND profiles.role = 'admin'
    )
);

-- Política: Un cliente solo puede ver SUS propios pedidos
CREATE POLICY "ver_mis_pedidos"
ON orders
FOR SELECT
USING (user_id = auth.uid());
-- Solo devuelve filas donde el user_id coincide con el usuario autenticado

-- Política: Cajeros y admins pueden ver TODOS los pedidos
CREATE POLICY "cajeros_ven_todos_pedidos"
ON orders
FOR SELECT
USING (
    EXISTS (
        SELECT 1 FROM profiles
        WHERE profiles.id = auth.uid()
        AND profiles.role IN ('cashier', 'admin')
    )
);

-- Política: Un usuario autenticado puede crear sus propios pedidos
CREATE POLICY "crear_pedidos"
ON orders
FOR INSERT
WITH CHECK (
    auth.uid() IS NOT NULL     -- Debe estar logueado
    AND user_id = auth.uid()   -- No puede crear pedidos a nombre de otro
);
-- WITH CHECK = condición que deben cumplir los DATOS que se van a insertar
Tip: auth.uid() es la función mágica de RLS
auth.uid() es una función de PostgreSQL que Supabase inyecta. Devuelve el UUID del usuario que está ejecutando la consulta, extraído del JWT. Si el usuario no está autenticado, devuelve NULL. Con esto, cada usuario "ve" solo sus propios datos sin que tengamos que filtrar nada en el código.

5. Service Role Key vs Anon Key — Las dos llaves del castillo

Supabase te da dos claves para acceder a tu proyecto, y confundirlas puede ser un grave problema de seguridad.

Clave Nivel de acceso ¿Dónde usarla? ¿RLS la afecta?
anon key (pública) Usuario anónimo Frontend (puede verla el usuario) Sí — respeta todas las políticas
service_role key (secreta) Superadministrador Solo en el servidor/backend No — bypassa todo RLS
Regla de oro: La service_role key NUNCA va al frontend
La service_role key tiene acceso total a tu base de datos, saltándose todas las reglas de seguridad. Si la pones en el código del frontend, cualquier usuario malicioso que inspeccione el código puede robarla y tener acceso completo a todos tus datos. Guárdala solo en el archivo .env del servidor y nunca la incluyas en el repositorio de Git. 
// ============================================================
// FRONTEND (navegador del usuario - código visible para todos)
// ============================================================

// ✅ CORRECTO: usa la anon key (pública, protegida por RLS)
const supabaseClient = createClient(
    CONFIG.SUPABASE_URL,
    CONFIG.SUPABASE_ANON_KEY  // ← Esta puede estar en el código del frontend
);

// ✅ El usuario solo puede hacer lo que las políticas RLS permiten


// ============================================================
// BACKEND (servidor privado - código NO visible para el público)
// ============================================================

// backend/src/services/supabase.js
import { createClient } from '@supabase/supabase-js';

// ✅ CORRECTO: usa la service_role key desde variables de entorno
const supabaseAdmin = createClient(
    process.env.SUPABASE_URL,
    process.env.SUPABASE_SERVICE_ROLE_KEY  // ← Solo en el .env del servidor
);

// ✅ El backend puede hacer operaciones privilegiadas:
// - Crear perfiles al registrar usuarios
// - Ver todos los pedidos para el panel del cajero
// - Calcular estadísticas sin restricciones de RLS
export default supabaseAdmin;
Tip: El archivo .env nunca va a GitHub
El archivo .env donde guardas tus claves secretas debe estar en el .gitignore. En el repositorio solo va el archivo .env.example con los nombres de las variables pero sin los valores reales. Ejemplo:
SUPABASE_URL=https://tu-proyecto.supabase.co
SUPABASE_SERVICE_ROLE_KEY=tu-clave-aqui

6. El trigger de profiles — Crear el perfil automáticamente

Cuando un usuario se registra en Supabase Auth, se crea automáticamente un registro en la tabla interna auth.users. Pero nuestra tabla pública profiles también necesita un registro para ese usuario. ¿Cómo lo sincronizamos?

Con un trigger que se ejecuta automáticamente cada vez que Supabase crea un usuario nuevo.

-- Función que crea el perfil cuando se registra un nuevo usuario
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
    -- Insertar en profiles con el mismo id que auth.users
    INSERT INTO public.profiles (id, full_name)
    VALUES (
        NEW.id,                                    -- UUID del nuevo usuario
        NEW.raw_user_meta_data->>'full_name'        -- Nombre enviado en el signUp()
        -- El operador ->> extrae un campo de texto de un JSON
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
-- SECURITY DEFINER: la función se ejecuta con los permisos de quien la creó (admin)
-- Necesario porque el trigger se dispara desde auth.users (esquema privado)
-- y necesita insertar en public.profiles

-- Trigger: ejecutar la función DESPUÉS de cada INSERT en auth.users
CREATE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW
    EXECUTE FUNCTION handle_new_user();
Analogía: El formulario de bienvenida
Cuando un empleado nuevo firma su contrato (INSERT en auth.users), automáticamente el departamento de RRHH crea su expediente personal (INSERT en profiles). No depende de que nadie se acuerde de hacerlo manualmente; el proceso es automático y garantizado. Eso hace nuestro trigger.
Tip: SECURITY DEFINER explicado
Normalmente una función SQL se ejecuta con los permisos del usuario que la llama. Con SECURITY DEFINER, se ejecuta con los permisos del usuario que la creó (el administrador de la BD). Esto es necesario aquí porque el evento INSERT en auth.users ocurre en un esquema privado al que los usuarios normales no tienen acceso, pero la función necesita poder escribir en public.profiles.

7. Creando usuarios de prueba

Para probar el sistema necesitamos al menos tres usuarios con roles diferentes: admin, cajero y cliente.

1
En el dashboard de Supabase, ve a "Authentication" en el menú izquierdo.
2
Haz clic en la pestaña "Users". Verás la lista de usuarios registrados (vacía al inicio).
3
Haz clic en "Add user" y crea los tres usuarios:
• Email: admin@aroma.com — Contraseña: Admin1234!
• Email: cajero@aroma.com — Contraseña: Cajero1234!
• Email: cliente@aroma.com — Contraseña: Cliente1234!
4
Verifica en el "Table Editor" que en la tabla profiles se crearon automáticamente 3 filas (gracias al trigger). Todos deberían tener role = 'client' por defecto.
5
Ve al SQL Editor y ejecuta el siguiente script para asignar los roles correctos: 
-- Asignar rol de admin al usuario admin@aroma.com
UPDATE profiles
SET role = 'admin'
WHERE id = (
    SELECT id FROM auth.users WHERE email = 'admin@aroma.com'
);

-- Asignar rol de cajero al usuario cajero@aroma.com
UPDATE profiles
SET role = 'cashier'
WHERE id = (
    SELECT id FROM auth.users WHERE email = 'cajero@aroma.com'
);

-- cliente@aroma.com ya tiene role = 'client' por defecto, no necesita cambio

-- Verificar que los roles quedaron correctamente asignados
SELECT
    u.email,
    p.full_name,
    p.role,
    p.active
FROM profiles p
JOIN auth.users u ON p.id = u.id
ORDER BY p.role;
Advertencia: Cambiar roles solo desde el backend o SQL Editor
Nunca deberías tener una funcionalidad en el frontend que permita al propio usuario cambiar su rol. La asignación de roles es una operación de administrador que solo debe ocurrir desde el SQL Editor o desde un endpoint del backend protegido. Si usas RLS correctamente, un cliente no puede hacer UPDATE profiles SET role = 'admin' porque la política lo bloquea.

8. Políticas RLS completas del proyecto Aroma

Este es el script completo de todas las políticas que necesita Aroma. Ejecuta este bloque en el SQL Editor de Supabase después de activar RLS en todas las tablas.

-- =========================================================
-- PASO 1: Activar RLS en todas las tablas
-- =========================================================
ALTER TABLE categories  ENABLE ROW LEVEL SECURITY;
ALTER TABLE products    ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders      ENABLE ROW LEVEL SECURITY;
ALTER TABLE order_items ENABLE ROW LEVEL SECURITY;
ALTER TABLE profiles    ENABLE ROW LEVEL SECURITY;


-- =========================================================
-- CATEGORIES
-- =========================================================

-- Cualquier visitante (sin cuenta) puede ver las categorías activas
-- Necesario para mostrar el menú en la página pública
CREATE POLICY "categorias_publicas"
ON categories FOR SELECT
USING (activo = true);

-- Solo admin puede crear, editar y eliminar categorías
CREATE POLICY "admin_gestiona_categorias"
ON categories FOR ALL
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'
);


-- =========================================================
-- PRODUCTS
-- =========================================================

-- Visitantes pueden ver productos activos (el menú)
CREATE POLICY "productos_publicos"
ON products FOR SELECT
USING (activo = true);

-- Admin puede ver TODOS los productos (incluyendo los desactivados)
CREATE POLICY "admin_ve_todos_productos"
ON products FOR SELECT
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'
);

-- Solo admin puede crear, editar y eliminar productos
CREATE POLICY "admin_gestiona_productos"
ON products FOR ALL
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'
);


-- =========================================================
-- ORDERS
-- =========================================================

-- Un cliente ve únicamente sus propios pedidos
CREATE POLICY "cliente_ve_sus_pedidos"
ON orders FOR SELECT
USING (user_id = auth.uid());

-- Cajeros y admins pueden ver TODOS los pedidos
CREATE POLICY "staff_ve_todos_pedidos"
ON orders FOR SELECT
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) IN ('cashier', 'admin')
);

-- Un usuario autenticado puede crear pedidos (solo los suyos)
CREATE POLICY "cliente_crea_pedidos"
ON orders FOR INSERT
WITH CHECK (
    auth.uid() IS NOT NULL       -- Debe estar logueado
    AND user_id = auth.uid()     -- Solo puede crear pedidos a su nombre
);

-- Cajeros y admins pueden actualizar el estado de los pedidos
CREATE POLICY "staff_actualiza_pedidos"
ON orders FOR UPDATE
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) IN ('cashier', 'admin')
);

-- Solo admin puede eliminar pedidos (operación muy restringida)
CREATE POLICY "admin_elimina_pedidos"
ON orders FOR DELETE
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'
);


-- =========================================================
-- ORDER_ITEMS
-- =========================================================

-- Un usuario puede ver los items de sus propios pedidos
CREATE POLICY "cliente_ve_sus_items"
ON order_items FOR SELECT
USING (
    order_id IN (
        SELECT id FROM orders WHERE user_id = auth.uid()
    )
);

-- Staff puede ver todos los items
CREATE POLICY "staff_ve_todos_items"
ON order_items FOR SELECT
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) IN ('cashier', 'admin')
);

-- Un usuario autenticado puede insertar items en sus pedidos
CREATE POLICY "cliente_crea_items"
ON order_items FOR INSERT
WITH CHECK (
    order_id IN (
        SELECT id FROM orders WHERE user_id = auth.uid()
    )
);


-- =========================================================
-- PROFILES
-- =========================================================

-- Cada usuario puede ver y editar solo su propio perfil
CREATE POLICY "ver_mi_perfil"
ON profiles FOR SELECT
USING (id = auth.uid());

-- Admin puede ver todos los perfiles (para el panel de usuarios)
CREATE POLICY "admin_ve_todos_perfiles"
ON profiles FOR SELECT
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'
);

-- Cada usuario puede actualizar SU perfil (nombre, teléfono)
-- pero no puede cambiar su propio rol
CREATE POLICY "actualizar_mi_perfil"
ON profiles FOR UPDATE
USING (id = auth.uid())
WITH CHECK (
    id = auth.uid()
    -- Nota: para evitar que el usuario se cambie el rol,
    -- implementa lógica adicional en el backend
);

-- Admin puede actualizar cualquier perfil (cambiar roles, desactivar)
CREATE POLICY "admin_actualiza_perfiles"
ON profiles FOR UPDATE
USING (
    (SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'
);
Tip: Puedes probar las políticas desde el Table Editor
En Supabase, el Table Editor normalmente actúa con la service_role key (bypassa RLS). Para probar las políticas como si fueras un usuario específico, usa el SQL Editor y ejecuta: SET LOCAL role TO authenticated; antes de tu consulta. También puedes usar el cliente de Supabase en el navegador con la anon key y hacer login con uno de tus usuarios de prueba.

9. Resumen del flujo de seguridad en Aroma

Veamos cómo todas las piezas encajan cuando un cliente hace un pedido:

Cliente hace pedido:
  1. Frontend envía email + password → Supabase Auth
  2. Supabase devuelve JWT con {user_id, role: "client"}
  3. Frontend guarda JWT en localStorage

Frontend crea el pedido:
  4. Frontend llama a Supabase con anon key + JWT en el header
  5. Supabase extrae user_id del JWT → auth.uid() = "uuid-del-cliente"
  6. RLS verifica política "cliente_crea_pedidos":
      auth.uid() IS NOT NULL ← true (está logueado)
      user_id = auth.uid() ← true (el pedido es suyo)
  7. Pedido creado exitosamente

Cajero actualiza el estado:
  8. Cajero logueado → su JWT tiene role: "cashier"
  9. Cajero llama UPDATE en orders con su JWT
  10. RLS verifica "staff_actualiza_pedidos":
      role IN ('cashier', 'admin') ← true
  11. Estado actualizado a "preparing"

Ataque frustrado:
  12. Cliente malicioso intenta leer pedidos de otros:
      SELECT * FROM orders;
  13. RLS solo devuelve filas donde user_id = auth.uid()
  14. Cliente solo ve SUS propios pedidos. Seguro.

Resumen de la Guía 10

  • Autenticación verifica quién eres. Autorización verifica qué puedes hacer. Son conceptos diferentes y complementarios.
  • Supabase Auth gestiona todo el ciclo de vida de los usuarios: registro, login, tokens, expiración, refresh. Nunca guarda contraseñas en texto plano.
  • Un JWT es un token con información del usuario firmado digitalmente. Es legible pero infalsificable. Contiene el user_id, email, rol y fecha de expiración.
  • RLS (Row Level Security) añade políticas de seguridad a nivel de fila en PostgreSQL. Con auth.uid() podemos hacer que cada usuario solo vea sus propios datos.
  • La anon key es pública (va en el frontend, protegida por RLS). La service_role key es secreta (solo en el servidor, bypassa RLS).
  • El trigger handle_new_user crea automáticamente el perfil en profiles cada vez que un usuario se registra en auth.users.
  • Los roles (admin, cashier, client) se guardan en profiles.role y las políticas RLS los consultan en tiempo real con cada operación.
  • Toda la seguridad del sistema depende de que estas capas funcionen juntas: JWT + RLS + políticas bien diseñadas.
