PARTE 1 – PREPARAR EL REPOSITORIO EN GITHUB
================================================================

PASO 1 – Crear el repositorio en GitHub
  - Ir a github.com -> New repository
  - Name: cafeteria-aroma (o el nombre que prefieras)
  - Private o Public: tu elección
  - NO marques "Initialize with README" (ya tienes uno)
  - Clic en "Create repository"

PASO 2 – Verificar que .gitignore protege los secretos
  Asegúrate de que .gitignore incluye:
    node_modules/
    .env
    backend/.env

  Si ya están ignorados, no hay riesgo de subir claves secretas.

PASO 3 – Hacer commit de todos los cambios pendientes
  git add .
  git status  // verificar qué archivos se agregarán

  ¡IMPORTANTE! Revisa que NO aparezcan:
  - backend/.env
  - backend/node_modules/
  Si aparecen, agrégatelos al .gitignore antes de continuar.

  git commit -m "feat: proyecto completo listo para produccion"

PASO 4 – Conectar con GitHub y subir
  GitHub mostrará estos comandos al crear el repositorio:
  (copia exactamente los que te da GitHub, no los de abajo)

  git remote add origin https://github.com/TU_USUARIO/cafeteria-aroma.git
  git branch -M main
  git push -u origin main

PASO 5 – Verificar en GitHub
  Abre tu repositorio en el navegador.
  ¿Ves todos los archivos excepto .env y node_modules?

PARTE 2 – DESPLEGAR EL BACKEND EN RAILWAY
================================================================

Railway es una plataforma que despliega Node.js fácilmente.
Tiene capa gratuita para proyectos pequeños.

PASO 1 – Crear cuenta en Railway
  - Ir a: https://railway.app
  - "Login with GitHub" -> autorizar Railway

PASO 2 – Crear nuevo proyecto
  - Clic en "New Project"
  - Seleccionar "Deploy from GitHub repo"
  - Buscar y seleccionar tu repositorio

PASO 3 – Configurar el servicio
  Railway detectará automáticamente que es un proyecto Node.js.
  Necesitas indicar que el código del backend está en una subcarpeta:
  - Settings -> Root Directory: backend
  - O en el panel de configuración:
    Start Command: npm start
    Build Command: npm install

PASO 4 – Configurar variables de entorno
  En Railway -> Variables -> Add Variables:

  SUPABASE_URL              = https://XXXX.supabase.co
  SUPABASE_SERVICE_ROLE_KEY = eyJh...tu-service-role-key...
  FRONTEND_URL              = https://tu-cafeteria.netlify.app
    (pondremos la URL real de Netlify en el siguiente paso)
  PORT                      = 3000
    (Railway asigna automáticamente el PORT, pero déjalo por si acaso)

  IMPORTANTE: Nunca pongas estas claves en el código fuente.
  Railway las inyecta como process.env.* en tiempo de ejecución.

PASO 5 – Deploy
  Railway inicia el deploy automáticamente.
  Progreso: Building -> Deploying -> Active

  Al completar, Railway da una URL como:
    https://cafeteria-aroma-production.up.railway.app

PASO 6 – Verificar el health check
  Abre en el navegador:
    https://cafeteria-aroma-production.up.railway.app/health

  ¿Responde {"status":"ok","timestamp":"..."}?

  Si hay errores:
  - Railway -> Logs: ver los mensajes de consola del servidor
  - Verifica que las variables de entorno estén correctas

PARTE 3 – DESPLEGAR EL FRONTEND EN NETLIFY
================================================================

Netlify despliega sitios web estáticos (HTML/CSS/JS) gratis.

PASO 1 – Crear cuenta en Netlify
  - Ir a: https://netlify.com
  - "Sign up with GitHub"

PASO 2 – ANTES de desplegar: actualizar config.js

  Abre frontend/js/config.js y actualiza la URL del backend:

    const CONFIG = {
      SUPABASE_URL:      'https://XXXX.supabase.co',   // sin cambios
      SUPABASE_ANON_KEY: 'eyJh...',                     // sin cambios
      API_BASE_URL:      'https://tu-url-railway.up.railway.app',  // ACTUALIZAR
    };

  Guarda, haz commit y push:
    git add frontend/js/config.js
    git commit -m "chore: URL del backend en produccion"
    git push

PASO 3 – Importar el proyecto en Netlify
  - Netlify -> "Add new site" -> "Import an existing project"
  - Conectar GitHub -> seleccionar el repositorio
  - Configuración:
    Base directory:    frontend
    Build command:     (dejar vacío, no necesita compilación)
    Publish directory: frontend

PASO 4 – Deploy
  Netlify despliega automáticamente.
  Al completar, da una URL como:
    https://cafeteria-aroma-XXXX.netlify.app

  Puedes personalizar el nombre: Site settings -> Site name.

PASO 5 – Actualizar FRONTEND_URL en Railway
  Ahora que tienes la URL de Netlify, actualiza en Railway:
  FRONTEND_URL = https://tu-cafeteria.netlify.app

  Railway re-desplegará automáticamente.

PASO 6 – Verificar CORS
  Abre la consola del navegador en tu sitio de Netlify.
  Haz cualquier petición: navega al menú.
  ¿Aparece el error "CORS policy"?

  Si hay error CORS:
  - Verifica que FRONTEND_URL en Railway sea exactamente la URL
    de Netlify (sin / al final, con https://).
  - En backend/src/index.js el cors() debe usar process.env.FRONTEND_URL

================================================================
PARTE 4 – CONFIGURAR DEPLOY CONTINUO
================================================================

Una de las ventajas de conectar GitHub a Railway y Netlify es
que el deploy es AUTOMÁTICO en cada push:

  git push -> Railway re-despliega el backend automáticamente
           -> Netlify re-despliega el frontend automáticamente

Flujo de trabajo en producción:
  1. Desarrollas localmente y pruebas.
  2. git add . && git commit -m "feat: nueva funcionalidad"
  3. git push
  4. Railway y Netlify detectan el push y re-despliegan.
  5. En ~2 minutos, la versión actualizada está en producción.

PARTE 6 – SOLUCIÓN DE PROBLEMAS COMUNES EN PRODUCCIÓN
================================================================

PROBLEMA: El frontend carga pero el menú está vacío.
  CAUSA: API_BASE_URL en config.js apunta a localhost.
  SOLUCIÓN: Actualizar config.js con la URL de Railway.

PROBLEMA: Error CORS en producción pero no en local.
  CAUSA: FRONTEND_URL en Railway no coincide con la URL de Netlify.
  SOLUCIÓN: Copiar exactamente la URL de Netlify a FRONTEND_URL.

PROBLEMA: Login falla con "invalid key".
  CAUSA: SUPABASE_ANON_KEY en config.js es incorrecto.
  SOLUCIÓN: Copiar de Supabase -> Settings -> API -> anon key.

PROBLEMA: Backend responde 500 en todos los endpoints.
  CAUSA: SUPABASE_URL o SUPABASE_SERVICE_ROLE_KEY incorrecto.
  SOLUCIÓN: Revisar las variables en Railway -> Variables.
  DIAGNÓSTICO: Railway -> Logs para ver el error real.

PROBLEMA: El comprobante no se sube (Storage error).
  CAUSA: El bucket no existe o no tiene política de escritura.
  SOLUCIÓN: Verificar en Supabase -> Storage y ejecutar el SQL
    de la política (Guía 13).

PARTE 7 – ALTERNATIVAS DE DESPLIEGUE
================================================================

BACKEND (alternativas a Railway):
  Render:   https://render.com
    - Free tier con 750 horas/mes.
    - Se "duerme" después de 15 min de inactividad (free tier).
    - Root Directory: backend
    - Build Command: npm install
    - Start Command: npm start

  Fly.io:   https://fly.io
    - Requiere instalar la CLI (flyctl).
    - Más control pero más configuración.

FRONTEND (alternativas a Netlify):
  GitHub Pages:
    - Completamente gratis.
    - Solo para proyectos públicos (o con GitHub Pro).
    - Limitación: el path del sitio puede ser /repo-name/, no /.
    - No requiere build, solo archivos estáticos.

  Vercel:   https://vercel.com
    - Similar a Netlify, también muy fácil.
